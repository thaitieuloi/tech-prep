Flink Core Concepts — Câu hỏi phổ biến trong PV

⏱ Event Time vs Processing Time

Q: Pipeline này dùng Event Time hay Processing Time? Tại sao?
Dùng Event Time. Timestamp được extract từ event.getTime() (log timestamp thực tế), không phải thời gian Flink nhận được. Điều này đảm bảo window tính toán đúng ngay cả khi dữ liệu đến trễ (network lag, backpressure).
Q: forBoundedOutOfOrderness là gì?
Watermark strategy cho phép event đến trễ tối đa N giây sau event mới nhất. Watermark = max_seen_timestamp - N. Ở đây N = allowedLatenessSec (default 30s). Sau watermark này, event sẽ bị coi là "late".

🪟 Tumbling vs Session Window

Tumbling (Metrics)

  • Fixed-size, non-overlapping
  • 5 phút một window
  • Key: orgId
  • Đếm success/fail per org
  • Predictable, good for dashboards

Session (Brute Force)

  • Dynamic, gap-based
  • Gap = 2 phút inactivity
  • Key: Tuple2(orgId, ip)
  • Cluster failed attempts per IP
  • Ideal for behavioral patterns

📦 Side Outputs & DLQ Pattern

Q: Side Output dùng để làm gì?
Side output cho phép emit các loại dữ liệu khác nhau từ cùng 1 operator mà không làm phức tạp main stream. Ở đây có 2 loại: PARSE_ERROR_DLQ_TAG (String — raw JSON lỗi) và LATE_EVENT_DLQ_TAG (OcsfAuthentication — events đến sau watermark).
Q: allowedLateness và sideOutputLateData khác gì nhau?
allowedLateness = thời gian window vẫn còn "sống" sau watermark để nhận thêm late events (trigger thêm). sideOutputLateData = sau khi window đóng hoàn toàn, events còn trễ hơn sẽ bị route sang side output thay vì bị drop.

🔄 AggregateFunction vs ProcessWindowFunction

Q: Tại sao dùng cả hai?
AggregateFunction tính toán incremental (không giữ toàn bộ elements trong memory), kết hợp với ProcessWindowFunction để access WindowContext (lấy window.getStart(), window.getEnd()). Đây là pattern hiệu quả nhất: low memory + window metadata.
Q: merge() trong AggregateFunction khi nào được gọi?
Khi Flink cần merge 2 session windows (SessionWindows có thể merge khi events của 2 session gặp nhau). Với TumblingWindows, merge() ít khi dùng nhưng vẫn phải implement.

☁ Checkpointing & Fault Tolerance

Q: Checkpoint interval bao lâu? Tại sao?
60 giây (CHECKPOINT_INTERVAL_MS = 60_000). Balance giữa: overhead của checkpoint (snapshot state + barriers) và recovery time (replay từ last checkpoint). Với security analytics, 60s acceptable.
Q: Kafka offset được quản lý thế nào?
Flink Kafka Source tích hợp với checkpoint — offset được commit vào Flink state, không phải Kafka consumer group. Điều này cho phép exactly-once semantics (với appropriate Kafka sink config). Dùng OffsetsInitializer.earliest() để consume lại từ đầu khi có state mới.

🏗 Architecture Patterns trong code này

Q: Multi-tenant design?
Toàn bộ topics, indexes, groupId đều có {orgId} suffix. DynamicKafkaSink route per-record dựa trên element.getMetadata().getOrgId(). Một job xử lý nhiều org đồng thời, mỗi org có data isolated.
Q: Idempotent writes OpenSearch?
DocId = sha256(orgId + "|" + time + "|" + ip). Khi Flink retry (at-least-once), cùng event sẽ write cùng docId → OpenSearch upsert → không tạo duplicate. Guard thêm: validate Instant.parse(time) trước khi index, tránh bulk API failure làm ảnh hưởng toàn checkpoint.

🔑 KeySelector & Parallelism

Q: Tại sao Branch B key bằng Tuple2(orgId, ip) thay vì chỉ ip?
Multi-tenant: IP 192.168.1.1 của org A và org B là hoàn toàn khác nhau. Nếu chỉ key by IP, brute force detection sẽ cross-contaminate giữa các org. Key compound Tuple2(orgId, ip) đảm bảo isolation.
Q: "unknown_ip" fallback ảnh hưởng gì?
Tất cả events không có IP sẽ group vào 1 key (orgId, "unknown_ip") — tạo ra một "super session" có thể trigger false positive nếu có nhiều lỗi auth không có IP. Đây là điểm có thể cải thiện (drop hoặc separate bucket).

⚠ Điểm cần cải thiện / Trade-offs

  • ⚠ printToErr() cho parse errors — không phù hợp production, nên dùng Kafka DLQ topic thực sự
  • ⚠ Chỉ xử lý 1 org (DEMO_ORG_ID) — cần extend để consume nhiều topics pattern
  • ⚠ unknown_ip fallback có thể tạo false positive brute force alerts
  • ⚠ OpenSearch sink: AT_LEAST_ONCE, sha256 dedup handle duplicate nhưng cần test kỹ bulk failure scenario
  • ✅ Tất cả magic strings đã được extract thành constants — dễ maintain
  • ✅ MAPPER là static field thread-safe (ObjectMapper)
  • ✅ NormalizerRegistry là transient — tránh serialization issue trong distributed env
