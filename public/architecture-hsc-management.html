System Overview

WHAT IS HSC PLATFORM?

HSC (Công ty Chứng khoán Hồ Chí Minh) is one of Vietnam's leading securities companies. The platform is an enterprise-grade Transaction & Trading system handling eKYC customer onboarding, fund deposit (nộp tiền), fund withdrawal (rút tiền), and securities trading — with complex approval workflows orchestrated by Camunda BPM.

Built on Spring Boot + JHipster microservices, event-driven via Apache Kafka, with Camunda orchestrating business-critical approval flows. The system was optimized to handle 10,000 transactions/sec — a 12× throughput improvement led by the candidate. Deployed on Azure with Jenkins CI/CD.

CORE MODULES
eKYC Onboarding Nộp tiền (Deposit) Rút tiền (Withdraw) Camunda Workflow Transaction Engine Audit & Compliance

Three business-critical flows, each orchestrated as a Camunda BPMN process: eKYC handles customer identity verification; Deposit handles incoming fund processing with bank reconciliation; Withdrawal handles fund release with multi-level approval.

Performance Optimization — 10,000 TPS

OPTIMIZATION JOURNEY — ~800 TPS → 10,000 TPS
01
Database — Index + Query Optimization
JMeter load tests + PostgreSQL EXPLAIN ANALYZE revealed N+1 queries and sequential scans on high-cardinality columns. Added composite indexes on (account_id, created_at), (status, processed_at). Switched reporting queries to read replicas. Rewrote N+1 patterns to batch JPA queries. Result: DB query p99 dropped from 800ms → 40ms.
02
Caching Layer — Redis
Account balance reads were hitting PostgreSQL on every transaction validation — highly frequent, low mutation rate. Introduced Redis cache with write-through strategy keyed by account:{id}:balance. Cache invalidated on any balance-changing event. Also cached static reference data (bank codes, product configs). Cache hit rate reached 85%, removing 85% of read load from PostgreSQL.
03
Async Offloading — Kafka
Non-critical operations (send SMS notification, write audit log, update analytics) were blocking the synchronous transaction path. Moved all to Kafka async consumers. Transaction Service now only does: validate → lock → update ledger → publish event → return. Notification and audit happen async. This removed ~200ms from every transaction's critical path.
04
Concurrency — Optimistic Locking + Connection Pool
Original pessimistic DB locks caused thread contention under concurrent load. Switched to optimistic locking (JPA @Version field on Account entity) — allows concurrent reads, retries only on conflict (rare in practice). Tuned HikariCP connection pool: max pool size calibrated to PostgreSQL's max_connections / number of pods. Eliminated connection exhaustion that caused cascading timeouts.
05
JVM Tuning + Horizontal Scaling
Profiling revealed excessive GC pauses causing latency spikes. Tuned G1GC: set -XX:MaxGCPauseMillis=50, calibrated heap size based on JMeter steady-state memory usage. Deployed Transaction Service as horizontally scaled pods on Azure — load balanced by Spring Gateway. Final result: 10,000 TPS sustained with p99 latency under 200ms.

System Design Best Practices

🏗️
Microservices via JHipster
JHipster generates consistent service scaffolding: Spring Security, Swagger, Liquibase migrations, CI config. Each service owns its domain and DB schema. No shared databases between services.
🔄
Camunda Externalized Workflow
Business flows in BPMN, not code. Hot-reload process definitions. Built-in audit, retry, escalation. Business team can adjust approval thresholds via DMN tables without code deployment.
⚡
SAGA Pattern
Choreography-based SAGA via Kafka for distributed transactions. No 2PC. Each service publishes events; compensating transactions handle rollback. Idempotency keys prevent double-processing.
🔐
Security by Design
JWT with short TTL. Input sanitization (SQL injection, XSS). PII fields masked in logs. Immutable audit logs for SSC compliance. Role-based access: customers cannot access each other's data.
📊
Full Observability
Correlation IDs across all services. Grafana dashboards: real-time TPS, Camunda process SLA, error rates. ELK for compliance audit queries. Alerts on TPS drop or error rate spike.
🧪
Test Pyramid
JUnit + Mockito for unit tests. Integration tests for service boundaries. E2E tests for full BPMN workflow validation. JMeter for TPS load simulation. SonarQube quality gates in CI.
📦
Outbox Pattern
Kafka events published atomically with DB writes via outbox table. Prevents event loss on service crash. Ensures "at-least-once" delivery with idempotent consumers.
🚀
CI/CD via Jenkins
Jenkins pipeline per microservice: build → unit test → SonarQube → integration test → deploy to Azure. Automated rollback on health check failure. Blue-green deployments for zero-downtime releases.

SYSTEM DESIGN & ARCHITECTURE
Q1
Walk me through the overall architecture of the HSC platform. How do eKYC, Deposit, and Withdrawal relate to each other?
Mô tả kiến trúc tổng thể của HSC. eKYC, Nộp tiền, Rút tiền liên quan nhau như thế nào?

Answer: HSC is a microservices platform where each business domain — eKYC, Deposit, and Withdrawal — is an independent Spring Boot service generated via JHipster. They share infrastructure (Kafka, PostgreSQL, Redis) but own separate DB schemas and APIs. The three domains are sequentially dependent from a business perspective: a customer must pass eKYC before they can deposit, and must have a positive balance before withdrawing — but each service enforces this independently via status checks, not shared DB joins.

The unifying layer is Camunda BPM, which orchestrates the workflow within each domain. When a customer submits a deposit, Deposit Service starts a Camunda process instance — Camunda drives the flow through risk check, approval, and bank API call. The services communicate business events asynchronously via Kafka (e.g., kyc.approved → Account Service activates; deposit.completed → Notification sends confirmation). This event-driven design means each service is fully autonomous and independently deployable.

📌 Key point: Camunda handles "what happens next in this business flow." Kafka handles "tell other services something happened." They serve different purposes and complement each other.

Q2
Why did you use JHipster? What concrete value did it bring to the team?
Tại sao dùng JHipster? Nó mang lại giá trị gì cụ thể cho team?

Answer: JHipster is a development platform that generates production-ready Spring Boot microservices with opinionated scaffolding. At HSC the team needed to ship multiple microservices quickly while maintaining consistency across them. JHipster gave us: (1) Standardized project structure — every service follows the same package layout, configuration pattern, and build script. Onboarding a new developer means they immediately understand any service. (2) Built-in security — OAuth2/JWT configuration, Spring Security setup generated correctly from day one. (3) Swagger/OpenAPI — every entity endpoint auto-documented. (4) Liquibase DB migrations — schema changes tracked and versioned. (5) CI/CD config — Jenkins pipeline file generated and ready.

The real value is not having to debate and decide these setup choices for each new service. The team focused on business logic instead of boilerplate configuration. Estimated saving: 2–3 days per new microservice for initial setup and documentation.

⚠️ Tradeoff: JHipster generates significant boilerplate — for very simple services it can feel over-engineered. We evaluated this case-by-case. Services with complex business logic (eKYC, Transaction) justified the full JHipster stack. Lightweight services (Notification) were simpler custom Spring Boot.

CAMUNDA BPM & WORKFLOW
Q3
Why Camunda for workflow orchestration? Could you have done the same with code (state machines)?
Tại sao chọn Camunda? Không thể dùng state machine trong code sao?

Answer: Yes, we could implement state machines in code — but Camunda offers critical advantages for a regulated financial system that a hand-coded state machine cannot easily replicate.

(1) Regulatory auditability: Camunda natively records every state transition with actor, timestamp, and input. SSC (State Securities Commission) requires a full traceable audit trail for every KYC decision and withdrawal approval. With Camunda this comes for free. With a custom state machine, we'd need to build and maintain this audit infrastructure separately — complex and error-prone.

(2) Business visibility: Compliance officers and managers can see exactly which Camunda process step a KYC application or withdrawal is at, in real-time via Camunda Cockpit — without reading code or querying databases. This was a concrete operational requirement at HSC.

(3) Business rule changes without deployment: When the approval threshold changed (auto-approve limit raised), the business analyst updated the BPMN diagram and DMN table. No code deployment. With a state machine, this is a code change + deployment + risk.

(4) Built-in resilience: Timer Boundary Events for escalation, Error Boundary Events for bank API failures, retry policies — all configured in BPMN without writing infrastructure code. The 50% processing time reduction came directly from replacing manual inter-department email hand-offs with automated Camunda task routing.

Q4
How did you design the Camunda External Worker model? Why not use embedded engine?
Thiết kế Camunda External Worker như thế nào? Tại sao không dùng embedded engine?

Answer: We used Camunda's External Worker pattern: the Camunda Engine runs as a separate service; our domain services (eKYC Worker, Transaction Worker) are stateless Spring Boot workers that poll the Camunda REST API for tasks, execute the business logic, then report completion back to Camunda.

Why not embedded engine? (1) Independent scaling: During peak KYC onboarding campaigns, we can scale OCR workers horizontally (more pods) without touching or redeploying the Camunda engine. With embedded engine, the engine scales with the service — wasteful and coupled. (2) Fault isolation: If a worker crashes mid-execution, the task remains in Camunda's queue — another worker picks it up on restart. Camunda manages retries automatically. (3) Technology flexibility: Workers can be any technology — our main workers are Java, but we could add a Python worker for AI processing if needed. (4) Centralized monitoring: One Camunda Cockpit gives visibility across all process types and all worker instances — impossible with multiple embedded engines.

📌 The external worker model is the production-recommended pattern for Camunda in microservices — it treats Camunda as the orchestration backbone, not an application dependency.

PERFORMANCE & SCALABILITY
Q5
How did you lead the team to achieve 10,000 TPS? What was your approach?
Bạn dẫn dắt team đạt 10,000 TPS như thế nào? Approach của bạn là gì?

Answer: The starting point was ~800 TPS with high p99 latency spikes. My approach was measure first, optimize second — never guess at bottlenecks. We set up JMeter load tests to reproduce the problem deterministically, and used Grafana + Prometheus to identify which service and which operation was the bottleneck.

The optimization came in layers: Database first — EXPLAIN ANALYZE revealed missing composite indexes and N+1 query patterns. Adding indexes and rewriting queries gave the biggest single gain (DB p99: 800ms → 40ms). Caching second — account balance reads were DB-heavy. Redis write-through cache removed 85% of read load. Async offloading third — non-critical operations (SMS, audit log) were blocking the transaction path. Moving to Kafka async removed ~200ms per transaction. Concurrency last — optimistic locking replaced pessimistic DB locks, and HikariCP pool tuning eliminated connection exhaustion.

As technical leader, I didn't do all the optimization myself — I identified the bottleneck layers, assigned ownership to team members, and ran weekly performance review sessions reviewing JMeter results against the previous baseline. The team saw measurable progress each sprint, which maintained motivation on what was a multi-sprint effort.

📌 Final: 10,000 TPS sustained, p99 < 200ms. Tools: JMeter (load), Grafana/Prometheus (metrics), PostgreSQL EXPLAIN (DB), JVM profiler (GC).

Q6
How do you handle distributed transactions across microservices? What pattern did you use?
Xử lý distributed transaction như thế nào? Dùng pattern gì?

Answer: We use the SAGA pattern (choreography-based) via Kafka. We explicitly avoided Two-Phase Commit (2PC) — it's fragile, doesn't scale, and holds DB locks across services. With SAGA: each step in a transaction publishes a Kafka event on success; downstream services react to that event. If any step fails, a compensating transaction event is published to undo previous steps.

For deposit: DebitExternalAccount → CreditInternalLedger → UpdateBalance → NotifyCustomer. If CreditInternalLedger fails → credit.failed event → triggers RefundExternalAccount. Every step is idempotent: services use unique transaction IDs and check for duplicate processing before acting. The Outbox Pattern ensures Kafka events are published atomically with DB writes — we write the event to an outbox table in the same DB transaction, then a background relay publishes to Kafka. This prevents the "DB committed but Kafka publish crashed" scenario.

⚠️ SAGA tradeoff: eventual consistency, not immediate. Between step 2 and step 3, the system is briefly inconsistent. For HSC's use cases (deposit processing is already async by nature), this was acceptable. For true atomic needs, we used DB-level transactions within a single service boundary.

SECURITY & COMPLIANCE
Q7
How did you design the eKYC system to be both secure and compliant with regulations?
Thiết kế eKYC system bảo mật và tuân thủ quy định như thế nào?

Answer: Security in eKYC operates at three layers. Data security: Customer identity documents (ID images, selfies) are never stored in our application DB — only metadata and verification results. Raw images go directly to Azure Blob Storage with server-side encryption, accessed only via pre-signed URLs with short TTL. Calls to external OCR/Face Match APIs use mutual TLS. PII fields are masked in application logs (log4j2 masking patterns).

Access control: Only compliance officers (specific role in JWT claims) can access eKYC applications in the Admin Portal. Spring Security method-level @PreAuthorize enforces this at the service layer — even if someone bypasses the UI. Customers can only see their own application status, never others' data.

Compliance: Camunda provides an immutable process history — every state transition (who reviewed, what decision, when) is recorded and cannot be modified. This directly satisfies SSC audit requirements. Additionally, the Audit Service writes a compliance-grade record to Elasticsearch with the full approval chain, which is indexed and queryable for regulatory inspection. All records have retention policies aligned with Vietnamese financial regulations.

Q8
How did you prevent SQL injection and XSS? What secure coding practices did you enforce?
Ngăn SQL injection và XSS như thế nào? Enforce secure coding ra sao?

Answer: SQL Injection: Completely prevented by using JPA/Hibernate with parameterized queries exclusively throughout the codebase. No string concatenation in queries anywhere. This was enforced as a SonarQube custom rule — any raw SQL string concatenation would fail the CI quality gate. JHipster's generated code uses Spring Data JPA by default, so developers had to actively fight against the framework to introduce a SQL injection vulnerability.

XSS: All user-supplied input is HTML-escaped before persistence using a custom sanitization filter applied at the controller input binding level. API responses always use Content-Type: application/json — not HTML — reducing XSS attack surface. For the Admin Portal frontend, React's default JSX escaping provides additional protection.

General secure coding: Spring Security CSRF protection enabled on all state-changing endpoints. JWT tokens with 15-minute access TTL and single-use refresh rotation. Dependency vulnerability scanning (OWASP Dependency Check) integrated in Jenkins pipeline — builds fail if high-severity CVEs are detected in dependencies. Code reviews had a security checklist: reviewers specifically checked for injection, authentication bypass, and insecure direct object reference patterns.

TECHNICAL LEADERSHIP
Q9
As Technical Leader at HSC, how did you estimate and plan complex projects involving multiple services?
Với vai trò Tech Lead tại HSC, bạn estimate và plan project phức tạp liên quan nhiều service như thế nào?

Answer: My estimation approach starts with breaking down work into dependency layers. For a feature like Camunda-based withdrawal: first identify the dependency tree — Camunda process design → External Worker implementation → Bank API integration → Notification → Audit. Each layer can be estimated independently and has a clear interface contract (BPMN process, OpenAPI spec) that allows parallel development once the contract is agreed.

I used three-point estimation (optimistic / realistic / pessimistic) for uncertain items, especially new technology integrations. For Camunda integration, we did a 3-day spike first — the spike output (working prototype + lessons learned) made the actual estimation much more accurate. Without the spike, we would have under-estimated by 40%.

I worked with leaders to negotiate scope trade-offs: the first sprint delivered a working Camunda flow with auto-approve only (no manual review). Manual review User Tasks came in sprint 2. This incremental delivery let us get feedback on the core flow early and reduced risk of building something wrong. Technical documentation — sequence diagrams, API contracts, DB schema — was produced before coding started, as a forcing function for teams to agree on interfaces before writing code.

Q10
How did you establish and enforce code quality standards for the team?
Bạn thiết lập và enforce code quality như thế nào cho team?

Answer: Code quality at HSC was enforced through systemic constraints, not just social agreement. The most impactful change was integrating SonarQube quality gates into Jenkins CI — a PR cannot be merged if: code coverage drops below threshold, new critical/blocker issues are introduced, or duplicated code exceeds limit. This made quality a technical requirement, not a style preference.

Code review process: Every PR required at least one review from a senior developer. I created a code review checklist covering: business logic correctness, security patterns (no SQL concatenation, input sanitization), test coverage for new paths, and adherence to the team's architecture decisions. I personally reviewed PRs touching cross-service contracts or Camunda process definitions — these were the highest-risk change areas.

Tech debt management: We tracked debt items in Jira like features — with impact assessment. Each sprint allocated 15–20% capacity for refactoring. This was non-negotiable even under deadline pressure — I had standing agreement with product management that this capacity existed. The investment paid off: when we needed to optimize for 10,000 TPS, the codebase was clean enough to refactor confidently without fear of hidden side effects.

Q11
How did you handle the integration with external Bank/VNPay APIs? What failure scenarios did you plan for?
Integrate với Bank/VNPay API như thế nào? Xử lý failure scenarios ra sao?

Answer: External Bank/VNPay API integration is a high-risk integration because failures directly impact customer money. We designed for failure from the start. Resilience4j wraps every external API call: timeout (30s hard limit), retry (2 retries with exponential backoff for transient errors), and circuit breaker (opens if failure rate >50% over 10 calls — prevents cascading failures when the bank is down).

Idempotency: Every API call includes a unique idempotency key (our transaction ID). If we retry due to a timeout and the first call actually succeeded, the bank API returns the original response instead of processing again. This prevents double-charging customers — critical for financial correctness.

Camunda Error Boundary Event: Bank API failures trigger a Camunda Error Boundary Event on the Service Task. The BPMN defines the error handling path — in deposit, a bank failure means status → FAILED and a compensating notification to the customer. In withdrawal, a failure after funds were released triggers SAGA compensation (refund event). The Camunda process history records exactly which step failed and why — invaluable for customer dispute resolution.

📌 We also implemented a manual reconciliation process (daily batch job) that compares our internal ledger against the bank statement — catches any discrepancy that slipped through all automated checks.

Q12
If you were to redesign HSC from scratch today, what would you do differently?
Nếu redesign HSC từ đầu hôm nay, bạn sẽ làm gì khác đi?

Answer: A few things I'd reconsider with the benefit of hindsight:

(1) Camunda 8 (Zeebe) instead of Camunda 7: We used Camunda 7 with its embedded/shared engine. Camunda 8 (Zeebe) is built for horizontal scaling from the ground up — the engine itself scales. For 10,000 TPS with complex workflows, Zeebe's event-sourced engine would have been a better fit architecturally, even though the migration path from Camunda 7 is non-trivial.

(2) Event schema registry from day one: As Kafka topics grew, managing backward-compatible schema evolution became painful. I'd introduce Avro schemas + Confluent Schema Registry from the first topic. This enforces schema contracts between producers and consumers, preventing silent breaking changes.

(3) OpenTelemetry distributed tracing earlier: Debugging a slow transaction that spans Gateway → Deposit Service → Camunda Worker → Bank API → Kafka → Audit was painful without end-to-end trace IDs. We added tracing late. It should be Day 1 infrastructure — instrument once, gain visibility everywhere.

(4) CQRS for reporting/analytics: Transaction Service became burdened with both write (process transactions) and read (management reports, analytics) workloads. I'd introduce a separate read model — updated via Kafka events, optimized for query patterns — to keep the write path clean and high-performance.
