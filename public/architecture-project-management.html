System Overview

WHAT IS GLOHOW?

Glohow is an Enterprise B2B SaaS platform that unifies Project Management (task, sprint, workflow) with an AI-powered Email Inbox. It enables enterprise teams to manage work in one place while using AI to translate, summarize, and suggest email responses — breaking language barriers in global cross-functional teams.

Built on a microservices architecture (Spring Boot), event-driven via Apache Kafka, with AI capabilities powered by RAG (Retrieval-Augmented Generation) using Qdrant vector database and multi-model LLMs (ChatGPT, Gemini, Claude, Ollama — configurable per tenant). Exposed via REST (OpenAPI/Swagger) and GraphQL APIs, deployed on AWS with full CI/CD via Jenkins.

MODULE 1 — PROJECT MANAGEMENT
Task / Ticket Sprint / Kanban Workflow Engine Multi-tenant

Full-featured PM module: projects, epics, tasks, subtasks, sprints, Kanban boards, assignees, due dates, comments, attachments, and role-based permissions per workspace. AI suggests task descriptions and auto-summarizes long comment threads.

MODULE 2 — AI EMAIL INBOX
Translation Summarization AI Suggest Reply RAG Context

Integrated email inbox linked to project context. AI auto-translates emails to user's preferred language, generates smart summaries, and suggests contextual reply drafts using project history as RAG knowledge base.

01
Email Ingestion & Pre-processing
User opens or receives email in Glohow inbox. Email Service stores raw content in PostgreSQL (status: RECEIVED). Detects language via lightweight classifier. Publishes email.ai.request event to Kafka with payload: {emailId, tenantId, targetLang, requestedFeatures: [TRANSLATE, SUMMARIZE, SUGGEST_REPLY]}.
02
RAG Context Retrieval (Qdrant)
AI Pipeline Worker reads from Kafka. Generates embedding of email content via embedding model. Queries Qdrant with top-k similarity search on tenant's collection (namespaced by tenantId). Retrieves: past similar emails, project context, glossary terms, user reply style profile.
03
Translation Engine — TM Lookup → Glossary Inject → LLM Call
Step A — Translation Memory (TM) lookup: Before any LLM call, the system queries the TM store (PostgreSQL, keyed by source-segment hash + tenantId). 100% match → return instantly, no LLM needed. Fuzzy match ≥75% → injected into prompt as a consistency reference.

Step B — Glossary injection: The tenant Glossary (Qdrant collection) is queried for every term in the email. Matched terms + their approved translations are injected as hard prompt constraints: "Always translate [source_term] as [approved_term]."

Step C — Multi-model LLM call: Final prompt = system instructions + TM references + Glossary constraints + RAG context + email body. Routed to the tenant-configured LLM provider (ChatGPT, Gemini, Claude, or Ollama). Returns structured output: translation, summary, suggestedReply[3 variants].
04
Result Persistence, Indexing & Caching
Results saved to PostgreSQL. Translation + reply embeddings indexed to Qdrant (continuous RAG learning). Full-text indexed to Elasticsearch. Results cached in Redis (TTL: 1h) keyed by email:{id}:ai:{version}. Status updated to AI_COMPLETE.
05
Push to Client via WebSocket
Notification Service consumes email.ai.completed Kafka event. Pushes result via WebSocket (STOMP) to the user's active session. If user is offline: result available on next load from Redis cache. User sees: translated email + summary badge + 3 AI reply suggestions with confidence indicators.
🗂️
Translation Memory (TM)
Every approved translation is stored as a TM segment: {sourceHash, sourceLang, targetLang, translatedText, tenantId, approvedAt}. Stored in PostgreSQL for exact/fuzzy match lookups.

100% match → served directly from TM, zero LLM cost.
Fuzzy match (75–99%) → TM suggestion injected into prompt as reference for the LLM to adapt.
No match → full LLM translation, result auto-saved to TM on user approval.

Result: repeated content (boilerplate, standard phrases) never hits the LLM again — cuts cost and ensures 100% consistency on approved segments.
📖
Glossary (Terminology DB)
Each tenant maintains a Glossary — a curated list of domain terms with approved translations: {sourceTerm, targetTerm, domain, tenantId, isMandatory}. Stored in Qdrant as a dedicated named collection for semantic term matching.

At translation time: email text is scanned for glossary terms (exact + semantic match via Qdrant). Matched terms are injected as hard constraints in the LLM prompt.

isMandatory=true → LLM must use the approved term (e.g., "Invoice" must always be "Hóa đơn", not "Biên lai").
Glossary is editable by workspace admins — changes take effect on the next translation immediately.
🔀
Multi-Model LLM Routing
LLM provider is configurable per tenant via the LLM Router service. Supported providers:

ChatGPT (GPT-4o) Gemini 1.5 Pro Claude 3.5 Sonnet Ollama (local)

Each model has a registered adapter implementing a common LLMPort interface — provider swap requires zero code change in the AI pipeline. Admin can configure: primary model, fallback model, and cost threshold (e.g., use Gemini Flash for short emails, GPT-4o for complex ones). Circuit breaker per provider; auto-failover to next in chain.
📋
Smart Summary
Generates 3-line executive summary for long emails. Extracts: key decisions, action items, deadlines. Linked to project context — if email mentions a task, summary includes task ID and current status.
💬
Reply Suggestion
Generates 3 reply variants: Formal / Neutral / Casual. Each grounded in RAG context (past replies by same user, project status). User can accept, edit, or regenerate. Accepted replies re-indexed to improve future suggestions.

System Design Best Practices

🏗️
Domain-Driven Microservices
Each service owns its domain, DB schema, and API contract. PM Service and Email Service are fully independent — can be deployed, scaled, and failed separately. No shared DB between services.
⚡
Async-First AI Pipeline
AI processing never blocks the API response. Kafka decouples request from processing — Email Service returns 202 immediately. AI results pushed via WebSocket when ready. Scales independently.
🧠
RAG over Fine-tuning
Domain knowledge (emails, tasks, glossary) lives in Qdrant, not model weights. Knowledge updates instantly without retraining. Tenant-specific context without data leakage. Explainable — we can trace every context chunk used.
🔐
Security by Design
JWT with short TTL + refresh rotation. Tenant isolation at every layer. Input sanitization (XSS, SQLi). Sensitive email data encrypted at rest. PII never sent to external LLM without consent flag.
🔄
Circuit Breaker for LLM
Resilience4j circuit breaker wraps all LLM calls. On cloud LLM failure → fallback to local Ollama. On total failure → return cached result or graceful degradation (show email without AI features).
📊
Full Observability
Structured logs with correlation IDs across services. AI-specific metrics: LLM latency p99, embedding time, cache hit rate, user acceptance rate. Grafana dashboards track AI quality over time.
🏢
Multi-tenant Architecture
Schema-per-tenant in PostgreSQL. Tenant-scoped Kafka topics. Isolated Qdrant collections. Zero cross-tenant data leakage. Tenant resolver middleware ensures correct context propagation throughout call chain.
🚀
CI/CD + Trunk-Based Dev
Jenkins pipelines per microservice. Unit + integration tests as gates. SonarQube quality gates block merges on quality regression. Feature flags for gradual AI feature rollout per tenant.

SYSTEM DESIGN & ARCHITECTURE
Q1
Walk me through the overall architecture of Glohow. How does the PM module integrate with the email/AI features?
Hãy mô tả kiến trúc tổng thể của Glohow. PM module tích hợp với email/AI feature như thế nào?

Answer: Glohow is a microservices-based B2B SaaS platform with two core modules sharing a common infrastructure. The PM Service manages the project graph — workspaces, projects, epics, tasks, sprints. The Email Service handles the inbox. Both are separate Spring Boot services, deployed independently, communicating via events through Apache Kafka.

The integration point is the AI Suggestion Service, which acts as a bridge. When a user views an email, the Email Service publishes a Kafka event. The AI pipeline processes it — using RAG with Qdrant to pull in relevant project context (the user's current tasks, project vocabulary, past replies). The AI's suggestions are therefore grounded in project reality, not generic.

Similarly, from PM side, when a user creates a task and wants an AI description, the PM Service calls the AI Suggestion Service, which retrieves similar past task descriptions and epic context from Qdrant. This creates a connected intelligence across both modules — users work faster because AI knows their project context.

📌 Key sell: The API layer exposes both REST (OpenAPI/Swagger) for service-to-service and GraphQL for the frontend — GraphQL lets the dashboard fetch project + tasks + AI summaries in one call instead of 4.

Q2
Why did you choose Kafka for the AI pipeline instead of direct REST calls?
Tại sao chọn Kafka cho AI pipeline thay vì gọi REST trực tiếp?

Answer: The core problem is that LLM calls are inherently slow — anywhere from 2–15 seconds depending on the model and prompt size. If we blocked the user's HTTP request on that, the UX would be terrible, and we'd exhaust our API thread pool under concurrent users.

Kafka solves this with the async request pattern: the Email Service publishes an event and immediately returns 202 Accepted with a job ID. The AI worker consumers process the event from the Kafka topic in the background. When done, a email.ai.completed event pushes the result to the user via WebSocket.

Beyond that, Kafka gives us: (1) Backpressure — if the LLM is slow, events queue up without dropping requests. (2) Replay — if the AI worker crashes mid-processing, we can replay from the last offset. (3) Independent scaling — we can scale AI workers independently from the Email Service based on queue depth (monitored via Prometheus + Grafana). (4) Audit trail — Kafka topics are durable logs of every AI request made.

⚠️ Tradeoff: Kafka adds operational complexity (broker management, topic design, consumer group coordination). For a startup in early stage, a simpler Redis queue might suffice. But for B2B enterprise with SLA requirements, Kafka's guarantees are worth it.

Q3
How did you design the multi-tenant architecture? How do you prevent data leakage between tenants?
Thiết kế multi-tenant như thế nào? Ngăn rò rỉ dữ liệu giữa các tenant ra sao?

Answer: We use a schema-per-tenant approach in PostgreSQL — each tenant gets their own schema (e.g., tenant_companyabc.tasks), giving hard database-level isolation while sharing a single PostgreSQL cluster. This is better than row-level isolation (where a missing WHERE clause could leak all tenants' data) and cheaper than database-per-tenant.

The Tenant Resolver middleware in the API Gateway reads the JWT's tenant claim, sets the correct schema in the database connection pool context, and propagates the tenant ID as a header throughout the call chain. Every downstream service respects this context — it's injected automatically via Spring's request scope.

For AI specifically: each tenant has an isolated Qdrant collection — their emails, tasks, and knowledge base never mix with another tenant's. Kafka topics are prefixed by tenantId. Redis keys are namespaced. This means even if a bug caused a query to hit the wrong collection, it would be a random tenant's collection — we added assertion checks at the AI worker level to prevent this.

📌 Key insight: The tenant isolation layer was designed upfront. Retrofitting multi-tenancy into an existing system is 10x harder — always build it in from day one.

AI / RAG ARCHITECTURE
Q4
Explain the RAG architecture in detail. Why RAG over just calling an LLM directly?
Giải thích chi tiết kiến trúc RAG. Tại sao dùng RAG thay vì gọi LLM trực tiếp?

Answer: RAG — Retrieval-Augmented Generation — is a pattern where we enrich the LLM prompt with retrieved relevant knowledge before generating a response, rather than relying on the model's static training data. In Glohow, this works alongside two other precision tools: Translation Memory (TM) and a Glossary.

Full translation pipeline: Before any LLM call, we first check the TM store (PostgreSQL) — if the source segment has been translated and approved before (100% match), we return that immediately. No LLM cost, perfect consistency. For fuzzy matches (≥75%), the TM result is injected into the prompt as a reference. Next, Glossary terms are extracted from the email and injected as hard constraints into the prompt. Only then does the LLM call happen — with: system instructions + TM references + Glossary constraints + RAG context chunks from Qdrant.

Multi-model architecture: The LLM provider is configurable per tenant. We implemented a LLMPort interface (Adapter pattern) with providers for ChatGPT (GPT-4o), Gemini 1.5 Pro, Claude 3.5 Sonnet, and Ollama (local). Swapping models requires zero pipeline code changes. Tenants can configure: primary model, fallback, and routing rules (e.g., use Gemini Flash for short emails under 200 tokens, GPT-4o for complex ones). Circuit breaker per provider with auto-failover.

Why this layered approach? TM = cost savings + consistency on repetition. Glossary = domain accuracy + terminology control. RAG = contextual relevance from live project data. LLM = generative intelligence for new content. Each layer has a job; none replaces the others.

📌 Metric: RAG reduced terminology inconsistency in translations by ~65% compared to zero-shot LLM calls in our internal testing.

Q5
How do you handle LLM failures or slow responses in production? What's your resilience strategy?
Xử lý LLM chậm hoặc lỗi như thế nào trong production? Chiến lược resilience là gì?

Answer: We implemented a multi-layer resilience strategy using Resilience4j. At the LLM call level, we have: (1) Timeout — 30s hard timeout per LLM call. (2) Retry — 2 retries with exponential backoff for transient failures. (3) Circuit Breaker — if failure rate > 50% over 10 calls, the circuit opens and we stop calling the failing provider for 60 seconds.

The fallback chain: Primary LLM slow/failing → fall back to next provider in the chain (e.g., ChatGPT → Gemini → Ollama) (faster, slightly lower quality). Ollama unavailable → serve from Redis cache (for identical or near-identical emails). Cache miss + all LLMs down → graceful degradation: show the email normally without AI features, with a "AI processing" badge. Never show an error screen — the core product (reading/sending emails) always works.

We monitor circuit breaker state in Grafana and alert on-call when the circuit opens. This was tested via chaos engineering — we randomly killed the cloud LLM endpoint in staging and verified the fallback chain worked within SLA.

Q6
How do you keep the AI suggestions improving over time? What's the feedback loop?
Làm thế nào để AI ngày càng cải thiện? Feedback loop hoạt động như thế nào?

Answer: The key insight is that user behavior is the training signal. When a user accepts an AI reply suggestion, edits it minimally, or marks a translation as correct — that's high-quality signal. When they regenerate or discard, that's negative signal.

Accepted outputs are re-embedded and indexed back into Qdrant with metadata {accepted: true, userId, editDistance: X}. Over time, the RAG retrieval naturally surfaces these high-quality accepted examples when similar emails come in. It's a continuous learning loop without retraining the model.

We also maintain prompt versioning — every prompt template is stored with a version ID. We A/B test new prompts against 10% of traffic and compare: acceptance rate, edit distance (lower = better), and user ratings. Grafana dashboards track these KPIs per tenant. A new prompt only gets promoted to 100% if it statistically outperforms the current one.

📌 This approach means the system gets more personalized per tenant over time — a company that translates many legal emails develops a legal-domain knowledge base that makes future translations sharper.

TECHNICAL DEPTH & PERFORMANCE
Q7
REST vs GraphQL — when did you use each, and what problems did GraphQL solve?
Khi nào dùng REST, khi nào dùng GraphQL? GraphQL giải quyết vấn đề gì?

Answer: We used both intentionally for different purposes. REST (OpenAPI) for: service-to-service communication (where contract predictability and HTTP caching matter), simple CRUD operations, webhook integrations, and file upload/download.

GraphQL for: frontend-facing complex data fetching. The classic problem it solved — the project dashboard originally required 4 REST calls: GET /projects/{id}, GET /projects/{id}/tasks, GET /tasks/{id}/ai-summary, GET /sprints/active. With GraphQL, one query projectDashboard(id) fetches all of this in a single round trip. This cut dashboard load time by ~40%.

We addressed GraphQL's classic problems: N+1 solved with DataLoader (batches DB calls per resolver). DoS via deep queries mitigated with query depth limiting and complexity scoring — queries exceeding complexity 100 are rejected. Versioning managed via deprecation annotations — old fields stay but are marked deprecated, giving clients time to migrate.

Q8
How did you implement secure coding practices? What vulnerabilities did you actively guard against?
Bạn áp dụng secure coding như thế nào? Bảo vệ khỏi các lỗ hổng nào?

Answer: Security was integrated into the development process, not bolted on after. Input validation: All user inputs are validated with Bean Validation annotations and a custom sanitization layer — HTML is escaped to prevent XSS before storage. GraphQL inputs go through custom scalar validators.

SQL Injection: Completely prevented by using JPA/Hibernate with parameterized queries exclusively. No raw SQL string concatenation anywhere in the codebase — enforced via SonarQube custom rules. Authentication: JWT tokens with 15-minute access token TTL and 7-day refresh rotation. Refresh tokens are single-use (revoked on use). Authorization: Spring Security method-level annotations (@PreAuthorize) enforce that users can only access their tenant's resources — even if the JWT is compromised, they can't escalate to another tenant.

AI-specific security: Email content sent to LLMs is stripped of PII (configurable per tenant) before the API call. Prompt injection attacks — where malicious email content tries to hijack the LLM prompt — are mitigated by clear prompt delimiters and input escaping.

TECHNICAL LEADERSHIP
Q9
As Technical Leader, how did you handle the decision to introduce AI/RAG into the product? How did you get team buy-in?
Với vai trò Tech Lead, bạn quyết định tích hợp AI/RAG vào sản phẩm như thế nào? Thuyết phục team ra sao?

Answer: The decision started with a clear problem statement: enterprise users were struggling with email communication across languages — a real pain point in B2B global teams. Before proposing RAG, I ran a 2-week spike with one developer to validate feasibility: could we build a working prototype with Ollama + Qdrant that produced noticeably better translations than zero-shot LLM calls?

The prototype validated it — domain-specific terminology retrieval improved consistency measurably. I then presented to leadership with: (1) Demo of the prototype with real emails, (2) Architecture proposal with effort estimate, (3) Risk assessment (LLM latency, cost, fallback strategy), (4) Phased rollout plan — ship basic translation first, add RAG improvement in phase 2.

Team buy-in came from involving senior developers early in the spike — they owned the architecture. I documented all decisions as ADRs (Architecture Decision Records) so junior developers understood the "why." We did knowledge sharing sessions on RAG concepts before implementation — nobody wants to build something they don't understand.

📌 Leadership lesson: Spikes de-risk technical bets. Always prove feasibility before committing sprint velocity to a new technology.

Q10
How did you ensure code quality and manage technical debt on a fast-moving feature team?
Làm thế nào đảm bảo chất lượng code và quản lý tech debt trong team phát triển nhanh?

Answer: Quality at speed requires systemic constraints, not just good intentions. We implemented four layers: (1) Automated gates in CI — Jenkins pipeline runs unit tests, integration tests, and SonarQube analysis on every PR. PRs can't be merged if coverage drops below threshold or if new critical/blocker issues are introduced. This makes quality a technical constraint, not a social one.

(2) Architecture governance — I conducted weekly architecture reviews where new patterns or dependencies were discussed. Architectural decisions were documented as ADRs. This prevented the "we just needed it quickly" anti-patterns that accumulate into debt. (3) Tech debt as a backlog citizen — every sprint had 15–20% capacity reserved for refactoring. We tracked debt items in Jira like any feature — with impact assessment and priority. (4) Pair programming and code reviews — every PR required at least one senior review. I personally reviewed PRs touching AI pipeline or architecture-sensitive code.

The result: we shipped features fast but kept code maintainable enough that new team members could onboard and contribute within a week.

Q11
How do you approach E2E testing for an AI feature? AI outputs are non-deterministic — how do you test that?
Kiểm thử E2E cho AI feature như thế nào? Output AI không ổn định — làm sao test?

Answer: Traditional assertion-based testing doesn't work for AI outputs — you can't assert that a translation is exactly a specific string. Instead, we test at three levels.

Level 1 — Pipeline integration tests: We test that the pipeline completes correctly — the Kafka event is consumed, the Qdrant query executes, the LLM is called with a well-formed prompt (captured via mock), and the result is stored and pushed via WebSocket. We mock the LLM, so the test is deterministic. This tests the "plumbing."

Level 2 — Output quality tests: We maintain a golden dataset of 50 emails with expected characteristics — "this email is in French, the translation should be in English and contain these 5 key terms." We run periodic batch evaluations against real LLM calls and score outputs using a rubric. Scores are tracked in Grafana — alerts fire if quality drops below threshold after a model upgrade.

Level 3 — Shadow testing: When we introduce new prompt versions, we run them in shadow mode (same input, new prompt, result not shown to user) and compare acceptance rate between old and new. Only promote if statistically better.

Q12
If you were to redesign Glohow from scratch today, what would you do differently?
Nếu redesign Glohow từ đầu hôm nay, bạn sẽ làm gì khác đi?

Answer: This question shows self-awareness, which is important for senior roles. A few things I'd reconsider:

(1) Event schema registry from day one: We didn't start with a Kafka schema registry (like Confluent Schema Registry). As the number of topics grew, managing backward compatibility of event schemas became painful. I'd introduce Avro schemas and a registry from the first event.

(2) CQRS for read-heavy endpoints: The project dashboard queries became complex joins as the product grew. I'd introduce a read model (CQRS pattern) — project read service with a denormalized PostgreSQL or MongoDB read store, updated via Kafka events. Simpler queries, better performance, decoupled from write path.

(3) OpenTelemetry earlier: We added distributed tracing late. Debugging a slow AI pipeline request that spans Email Service → Kafka → AI Worker → Qdrant → LLM was painful without proper trace IDs. OpenTelemetry should be Day 1 infrastructure.

(4) Embedding model versioning strategy: When we upgraded our embedding model, all existing Qdrant vectors were incompatible (different dimensions). We had to re-index everything. I'd design a migration strategy for embedding model upgrades upfront — dual-index with gradual cutover.

📌 Showing you can critically evaluate your own work demonstrates technical maturity — this is what senior/lead roles look for.
