Q1
How would you handle data consistency between Cassandra and PostGIS when a new hazard is confirmed? What if one write succeeds and the other fails? Làm thế nào để đảm bảo data consistency giữa Cassandra và PostGIS khi một hazard mới được xác nhận? Nếu một write thành công còn write kia thất bại?
▾
This is a classic distributed transaction problem. Since Cassandra and PostGIS are two separate stores, ACID transactions across both are impossible. My approach: Saga pattern with compensating transactions.

Step 1: Write telemetry to Cassandra first (append-only, low risk).
Step 2: Publish hazard.confirmed event to Kafka.
Step 3: Map Hazard Service consumes event → writes spatial point to PostGIS.
Failure handling: Kafka retains message → consumer retries with exponential backoff → max retries exceeded → Dead Letter Queue (DLQ). Hazard stays "PENDING" until resolved.

A periodic reconciliation job (every 5 min) cross-checks Cassandra vs PostGIS and re-queues missing spatial records — ensuring eventual consistency without blocking the real-time alert path.
Interview (Highlight Phỏng vấn): "We achieve eventual consistency without a distributed transaction coordinator. The Saga pattern trades atomicity for availability — correct trade-off for a system where writes must not block reads.
We achieve eventual consistency without a distributed transaction coordinator. The Saga pattern trades atomicity for availability — correct trade-off for a system where writes must not block reads.Chúng ta đạt trạng thái Nhất quán cuối (Eventual Consistency) mà không cần bộ điều phối (Coordinator) phân tán tốn kém. Mô hình Saga đánh đổi tính toàn vẹn tức thời (Atomicity) lấy tính khả dụng (Availability) — đánh đổi chính xác để Write không cản Read.
"
Vấn đề: Không thể dùng ACID cross-store. Giải pháp: Saga pattern — write Cassandra trước → publish event Kafka → service khác write PostGIS. Nếu fail: retry exponential backoff → DLQ sau max retry. Hazard giữ trạng thái PENDING. Reconciliation job 5 phút/lần so sánh 2 store và re-queue record thiếu. Đây là eventual consistency không block real-time path.
Q2

▾
Address at 3 levels simultaneously:

1. Context enrichment (pre-processing): Feed the model vehicle speed + road type from map tiles + known speed bump coordinates from OpenStreetMap. Filter known speed bump locations at pre-processing — never reach ML inference.

2. Runtime threshold tuning: Store confidence threshold in Redis as a hot-config key. Ops team raises threshold in real-time via admin API — no redeploy, effect in <1 second. Alert fires only above threshold.

3. Continuous retraining with crowd feedback: Driver dismissal → labeled false positive → nightly MLflow retrain. Champion/Challenger A/B test on 10% traffic. Auto-rollback if new model precision drops below SLA.
Interview (Highlight Phỏng vấn): "The threshold in Redis is the fastest lever — we can tune the model's behavior in production within seconds, without a deployment pipeline.
The threshold in Redis is the fastest lever — we can tune the model's behavior in production within seconds, without a deployment pipeline.Sử dụng Redis Threshold để tinh chỉnh AI Model Production là đòn bẩy phản ứng thần tốc — ngay lập tức chuyển đổi hành vi trí tuệ nhân tạo chỉ bằng config.

3 lớp: (1) Context enrichment: enrich data với tốc độ xe, loại đường, vị trí gờ giảm tốc từ OpenStreetMap → filter trước khi vào ML; (2) Threshold Redis hot-config: thay đổi ngưỡng runtime không cần deploy, có hiệu lực <1 giây; (3) Nightly retrain: feedback tài xế → label false positive → MLflow retrain → Champion/Challenger A/B 10% traffic → auto-rollback nếu precision giảm.

Q14
The business decides to offer a real-time hazard API to third parties (Google Maps, Waze). This introduces external consumers with SLA requirements you can't control. How do you design a public API layer without compromising your internal system's reliability? Business muốn mở public API cho bên thứ ba (Google Maps, Waze) để consume hazard data. Điều này đưa vào external consumer với SLA không kiểm soát được. Thiết kế public API layer như thế nào mà không ảnh hưởng độ tin cậy hệ thống nội bộ?
▾
Core risk: external consumers introduce unpredictable traffic patterns, and their failures must not cascade into internal systems.

① Bulkhead pattern — strict traffic isolation:
Public API runs on entirely separate infrastructure — separate K8s namespaces, separate API Gateway instances, separate rate limit counters. A Google Maps traffic spike cannot consume resources from internal fleet API. The bulkhead ensures external layer fails independently.

② Read-only materialized view — never expose internal DB:
External API consumes a pre-computed, read-only hazard snapshot updated every 30 seconds via a Kafka consumer writing to a dedicated read replica or Redis-backed hazard cache. External APIs never touch operational PostgreSQL/Cassandra. A DDoS on the public API only affects the read replica — internal writes are completely insulated.

③ API versioning + SLA tiers from day one:
Design v1/v2 from the start — never release v1 without planning v2. Define SLA tiers by partner class: Tier 1 (Google/Waze — 99.5% SLA, dedicated capacity), Tier 2 (smaller partners — 99%, shared pool), Tier 3 (free tier — best-effort, aggressive rate limits).

④ Push over poll — Kafka-backed webhooks:
Most third parties will poll every 5 seconds — catastrophically inefficient at scale. Instead: Kafka-backed webhook delivery — partners register a webhook endpoint and receive hazard events pushed within seconds. Premium partners get direct Kafka topic subscription via Confluent Cloud. Eliminates polling load entirely.

Q15
Your team has grown to 8 squads, each owning different microservices. Squad A changes a Kafka event schema and silently breaks Squad B's consumer — no error, just wrong data. How do you design a schema governance strategy that enables team autonomy without creating a distributed monolith? Team đã lớn lên thành 8 squad. Squad A thay đổi Kafka event schema làm break consumer của Squad B một cách âm thầm — không có lỗi, chỉ là dữ liệu sai. Thiết kế schema governance như thế nào để squad tự chủ mà không tạo ra distributed monolith?
▾
This is a sociotechnical governance problem — as much about human coordination as technology. The failure mode (silent wrong data) is more dangerous than an error because it's invisible.

① Schema Registry as the contract enforcer:
All Kafka producers must register schemas in Confluent Schema Registry. Registry enforces compatibility rules:
• BACKWARD: new schema reads old data — consumers don't update immediately.
• FORWARD: old schema reads new data — producers deploy before consumers.
• FULL: both directions — safest, most restrictive.
A producer attempting a breaking change (removing a required field) is rejected at publish time in CI/CD — never reaches production.

② Explicit version field in every event envelope:
Every event includes a schema_version field. Consumers switch on version, not on field presence. v1 and v2 events coexist on the same topic during migration. Consumer supports both for a defined deprecation window (4 sprints), then drops v1.

③ Consumer-Driven Contract Testing (CDCT — Pact):
Each consuming squad publishes a Pact contract defining exactly which fields they read. The producing squad's CI pipeline runs contract tests against all registered consumer contracts before merge. If Squad A's change breaks Squad B's Pact → CI fails, Squad A cannot merge without Squad B's explicit consent. Governance shifts left into the development workflow — not post-deployment firefighting.

④ Events as public API — AsyncAPI spec:
Teams must treat Kafka events as public API contracts, documented in AsyncAPI spec alongside OpenAPI. Changes require an RFC process reviewed by affected squads. The Platform team owns the Schema Registry as the final arbiter — not the producing squad.
