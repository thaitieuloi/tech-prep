01 Kafka đảm bảo message ordering như thế nào? Làm sao giữ ordering khi cần scale consumers?
Trung bìnhKafka+
▸ Đáp án gợi ý
// KAFKA ORDERING: PARTITION-LEVEL GUARANTEE
Kafka chỉ đảm bảo ordering trong cùng một partition. Messages có cùng key sẽ luôn đi vào cùng partition (nhờ hash(key) % num_partitions). Mỗi partition chỉ được đọc bởi 1 consumer trong 1 group tại một thời điểm.

Scale mà không mất ordering: tăng số partition + tăng số consumer song song, nhưng đảm bảo partition key được chọn đúng (ví dụ user_id cho user events). Không dùng null key nếu cần ordering.
Ứng viên senior phải biết: rebalance giữa consumers có thể tạo khoảng trống tạm thời — cần cooperative rebalancing (Kafka 2.4+) để minimize downtime.

▸ Đáp án gợi ý

Avro + Confluent Schema Registry là combination chuẩn. Schema Registry enforce compatibility mode trước khi producer publish.

Safe evolution rules (BACKWARD mode): chỉ ADD optional fields với default values, không xóa hoặc đổi tên field. Khi cần breaking change, dùng topic versioning (orders-events-v2) hoặc dual-write vừa write v1 vừa v2.

04 Kafka Consumer Lag là gì? Làm thế nào monitor và xử lý khi lag tăng đột biến?
Trung bìnhKafka+
▸ Đáp án gợi ý
// CONSUMER LAG: LOG-END-OFFSET - CONSUMER-OFFSET
Consumer Lag = log-end-offset − committed-offset. Lag tăng khi tốc độ produce > tốc độ consume.

Monitor: dùng kafka-consumer-groups.sh --describe, Confluent Control Center, hoặc export metric qua JMX → Prometheus → Grafana. Alert khi lag > threshold (ví dụ 10k messages hoặc lag > 5 phút).

Root causes thường gặp: slow downstream DB, GC pause, network bottleneck, expensive deserialization, hoặc spike traffic đột ngột.
SLA điển hình: lag < 30s cho near-realtime. Nếu lag tăng liên tục (không phải spike) → đây là capacity planning issue, không phải bug.
05 Thiết kế hệ thống replay event từ Kafka cho toàn bộ dữ liệu 6 tháng. Những thách thức gì cần giải quyết?
KhóKafka+
▸ Đáp án gợi ý
// KAFKA REPLAY STRATEGY — 6 MONTHS OF EVENTS
Best practice: archive raw events vào S3 (dưới dạng Avro/Parquet), sau đó dùng replay producer để đọc từ S3 → publish vào replay topic riêng → consumers đọc từ replay topic với consumer group riêng.

Phải giải quyết: throttling (tránh overwhelm), idempotency (tránh duplicate side-effects), và isolation (replay không ảnh hưởng live traffic).
Hỏi thêm: "Làm sao bạn đảm bảo downstream state (DB, cache) consistent sau replay?" — ứng viên giỏi sẽ nhắc đến CQRS, event sourcing, và idempotency keys.

▸ Đáp án gợi ý

// State TTL để tránh state bloat (Java) StateTtlConfig ttlConfig = StateTtlConfig .newBuilder(Time.hours(24)) .setUpdateType(StateTtlConfig.UpdateType.OnCreateAndWrite) .setStateVisibility(StateTtlConfig.StateVisibility.NeverReturnExpired) .build(); ValueStateDescriptor<Integer> descriptor = new ValueStateDescriptor<>("loginCount", Integer.class); descriptor.enableTimeToLive(ttlConfig);

08 Flink Checkpointing hoạt động như thế nào? Làm sao tune checkpoint để đạt exactly-once với latency thấp?
KhóFlink+
▸ Đáp án gợi ý
// CHANDY-LAMPORT ALGORITHM — FLINK CHECKPOINT FLOW
Flink dùng Chandy-Lamport algorithm: JobManager inject checkpoint barrier vào data stream. Khi operator nhận barrier, nó snapshot state của mình vào durable storage (S3/HDFS). Sau khi tất cả operators ack → checkpoint complete.

Exactly-once với Kafka sink: dùng Two-Phase Commit (2PC) — Flink pre-commit khi nhận barrier, commit thật sự sau khi checkpoint hoàn thành. Nếu fail giữa chừng → rollback về checkpoint cuối cùng.
Unaligned checkpointing (Flink 1.11+) giải quyết vấn đề backpressure làm chậm checkpoint — cho phép barrier vượt qua data đang chờ trong buffer.
09 Làm sao xử lý out-of-order events trong Flink? Watermark strategy nào phù hợp với từng use case?
Trung bìnhFlink+
▸ Đáp án gợi ý
// WATERMARK STRATEGIES — OUT-OF-ORDER EVENT HANDLING
Watermark = max_event_time_seen − max_out_of_order_duration. Flink trigger window khi watermark vượt qua window end time.

Events đến sau watermark (very late) sẽ bị discard hoặc xử lý bởi allowedLateness(). Với multi-partition sources, Flink lấy minimum của all partition watermarks → đảm bảo không trigger sớm.
Watermark quá lớn → latency cao. Watermark quá nhỏ → nhiều late events bị discard. Cần đo 99th percentile của event delay thực tế để chọn giá trị phù hợp.

19 Thiết kế hệ thống real-time fraud detection xử lý 1 triệu events/giây với latency <100ms. Kiến trúc đầy đủ?
KhóSystem Design+
▸ Đáp án gợi ý
// REAL-TIME FRAUD DETECTION — 1M events/s, <100ms latency
Key design decisions: 96 partitions Kafka (scale later), stateful Flink với RocksDB (user velocity counter per user_id), ML model embedded in Flink operator (tránh network hop), Redis cho feature lookup (<1ms).

Fault tolerance: Flink checkpoint mỗi 30s, Kafka replication factor=3, Redis cluster mode. Khi Flink job fail → replay từ Kafka offset.
Ứng viên phải tính toán được: 1M events/s × 96 partitions = ~10,400 events/partition/s. Flink parallelism = 96 để map 1:1 với Kafka partitions.

A02 Tại sao org_id được inject từ Kafka topic name thay vì đọc từ payload? Security implication là gì nếu làm ngược lại?

// ORG_ID INJECTION: TOPIC-BASED vs PAYLOAD-BASED

⚡ Part 2 — Flink Windowing & State 6 câu▼
B01 Tại sao bạn chọn event time thay vì processing time? Nếu dùng processing time, điều gì sai trong brute-force detection?
DễFlink+
▸ Đáp án gợi ý
// EVENT TIME vs PROCESSING TIME — BRUTE FORCE DETECTION FAILURE
Critical failure mode với processing time: Consumer lag của 3 phút khiến 10 failure events (xảy ra trong 9 giây) bị phân tán sang 2 different session windows. Alert không bao giờ fire — attacker thoát qua vì Flink "sees" events tại thời điểm khác với thực tế.

Event time dùng timestamp từ event.getTime() — timestamp do attacker's machine tạo ra, không bị ảnh hưởng bởi lag hay jitter trong pipeline.
B02 Giải thích cụ thể điều gì xảy ra khi một late event "bridges" 2 closed session windows trong brute-force detection. Code bạn handle điều này như thế nào?
KhóFlink+
▸ Đáp án gợi ý
// SESSION WINDOW MERGING: LATE EVENT BRIDGES TWO SESSIONS
Cơ chế: Khi late event đến trong allowedLateness window, Flink phát hiện nó "bridges" gap giữa 2 sessions. Flink merge 2 session states thành 1, re-emit kết quả merged. Result: Session1 (5 failures) + late event + Session2 (5 failures) = 11 failures → alert threshold 10 đạt được.

Vấn đề: Cả 2 sessions đã emit kết quả trước đó. Merged emit là lần thứ 3 → downstream nhận duplicate alerts.

Giải pháp trong code: Document rõ trong README rằng downstream consumers phải deduplicate trên (orgId, srcIp, windowStart). Nếu cùng windowStart nhận 2 alerts, chỉ giữ alert có failureCount cao nhất (latest merge result).
Đây là câu hỏi rất deep về Flink session semantics. Mention rằng Flink 1.18 hỗ trợ "late firing" behavior và bạn đã document điều này trong README.
B03 Tại sao bạn dùng static inner classes cho tất cả Flink operators thay vì lambda? Giải thích NotSerializableException.
Trung bìnhFlink+
▸ Đáp án gợi ý
// FLINK SERIALIZATION: LAMBDA vs STATIC INNER CLASS
Root cause: Java lambda và anonymous class đều ngầm capture this của outer class. Flink phải Java-serialize tất cả operators để gửi tới TaskManagers. Nếu outer class không implement Serializable, JVM throws NotSerializableException.

NormalizerRegistry được declare transient và recreate trong open() method — open() chạy trên mỗi TaskManager sau khi deserialization.
B04 Watermark = 30s và allowedLateness = 30s. Điều này có nghĩa là events tới trễ tối đa bao lâu vẫn được xử lý đúng window? Vẽ timeline.
KhóFlink+
▸ Đáp án gợi ý
// WATERMARK (30s) + ALLOWED LATENESS (30s) = 60s TOTAL TOLERANCE
Breakdown: Watermark = max_event_time_seen - 30s. Window [0,30) fires khi watermark ≥ 30s (tức là khi max event seen = 60s). Sau khi fire, window vẫn tồn tại thêm 30s (allowedLateness). Events có event_time thuộc [0,30) nhưng đến trong khoảng này sẽ re-trigger window emit. Events sau 60s hoàn toàn bị route to DLQ qua side output.
Điểm dễ nhầm: "60s total" là tính từ event time, không phải processing time. Watermark và lateness đều đo trên event time axis.
B05 Tại sao NormalizerRegistry được khai báo transient trong Flink operator và recreate trong open()? Nếu không làm vậy thì sao?
Trung bìnhFlink+
▸ Đáp án gợi ý
// TRANSIENT + open() PATTERN — FLINK OPERATOR LIFECYCLE
transient bảo Java serializer bỏ qua field này khi serialize operator. Khi TaskManager deserialize operator, field là null. open() được Flink gọi ngay sau deserialize — đây là nơi init lại mọi non-serializable resources (DB connections, in-memory caches, registries...).
B06 Nếu Flink job restart từ checkpoint sau khi OpenSearch sink crash, những events nào có thể bị re-processed? SHA-256 _id của bạn giải quyết điều này như thế nào?
KhóFlink+
▸ Đáp án gợi ý
// CHECKPOINT RESTART + IDEMPOTENT OPENSEARCH SINK
Scenario: Checkpoint T1 lưu Kafka offset=1000. Job crash sau khi process events 1001-1300 (một phần đã index). Restart từ T1 → Kafka offset reset về 1000 → events 1001-1300 được re-process.

Events 1001-1200 đã index sẽ bị re-index với cùng _id = SHA-256(orgId + event.getTime() + srcIp). Vì time và srcIp không thay đổi, _id giống hệt → OpenSearch upsert overwrites với cùng content → no duplicate documents.

C01 Trong bài answers.md, bạn đề xuất DLQ circuit breaker sau N lần OpenSearch fail. Làm sao phân biệt transient failure (529 rate limit) vs permanent failure (mapping error)? Xử lý khác nhau như thế nào?
KhóReliability+
▸ Đáp án gợi ý
// OPENSEARCH ERROR CLASSIFICATION: TRANSIENT vs PERMANENT
Classification: HTTP 5xx và 429 = transient → exponential backoff retry (1s, 2s, 4s, max 3 retries). HTTP 400 (mapping conflict, invalid field type) = permanent failure → no amount of retry sẽ fix → route to DLQ ngay lập tức + alert team.

Bug trong code hiện tại của bạn: OcsfOpensearchEmitter drop document khi timestamp unparseable — đây đúng cho permanent error. Nhưng nếu OpenSearch trả về 429, nên retry chứ không drop.
Đây là câu bạn có thể chủ động nêu ra: "Trong code submit, tôi handle timestamp parsing error nhưng chưa implement retry logic cho transient OpenSearch errors. Production cần AsyncSinkBase với built-in retry."
C02 Trong replay scenario (Part 3 Q4), bạn dùng consumer group riêng để reset offset. Vì sao không reset offset của production consumer group? Điều gì xảy ra với events đang được process khi reset?
KhóReliability+
▸ Đáp án gợi ý
// REPLAY WITHOUT DISRUPTING PRODUCTION: SEPARATE CONSUMER GROUP
Lý do cốt lõi: Production consumer group có associated Flink state (window accumulators, checkpoints). Reset offset của nó sẽ invalidate toàn bộ state — Flink không biết đâu là "ground truth" nữa. Hơn nữa, production consumers đang chạy realtime cho tất cả tenants, không chỉ org1.

Separate consumer group = separate offset tracking = separate Flink job → completely isolated. Production không bị ảnh hưởng 1 giây.
C03 Giải thích Flink credit-based backpressure. Điểm khác biệt với traditional blocking I/O là gì? Và tại sao nó tốt hơn cho streaming?
Trung bìnhReliability+
▸ Đáp án gợi ý
// CREDIT-BASED BACKPRESSURE: HOW IT PROPAGATES
Credit-based: Downstream operator grant "credits" (buffer slots) to upstream. Upstream chỉ send khi có credits. Khi sink chậm → không grant credits → upstream tự throttle → Kafka consumer slows naturally. Không có thread blocking, không OOM.

Traditional blocking I/O: slow downstream → upstream threads block → thread pool exhausted → OutOfMemoryError → job crash → checkpoint lost → data loss.
C04 Bạn phát hiện bug trong integration test: timestamp malformed gây OpenSearch rejection và Flink restart loop. Describe root cause và fix bạn đã apply.
DễDebug+
▸ Đáp án gợi ý
// BUG: MALFORMED TIMESTAMP → OS REJECTION → CHECKPOINT LOOP
Root cause: Shell loop with i going from 1-10, string interpolation produced 14:32:010.000Z (3-digit seconds) — invalid ISO-8601. OpenSearch strict_date_optional_time mapping rejected the document. The bulk indexer threw exception → Flink checkpoint flush failed → job restarted → replayed same event → infinite loop.

Key lesson: Validate at sink, not throw. A single bad document should not crash the entire checkpoint. Drop it, log it, continue. In production this should also go to DLQ.
Chủ động mention bug này trong interview thể hiện bạn thực sự chạy integration test và debug thật — không chỉ submit code chưa test.
C05 Bạn propose "per-tenant rate limiting với token bucket keyed by org_id trong Flink" ở phần What I Would Do Next. Implement điều này có ảnh hưởng gì tới windowing semantics không?
KhóDesign+
▸ Đáp án gợi ý
// PER-TENANT RATE LIMITING IN FLINK: WATERMARK IMPACT
Critical insight: Nếu rate limiter delay events của noisy tenant bằng cách buffering, những events đó sẽ "arrive late" vào Flink window operator theo event time perspective. Watermark đã advance → events bị classified là late → route to DLQ thay vì window.

Correct approach: Rate limiter phải đặt trước watermark assignment. Hoặc dùng DLQ-safe delay (drop + DLQ với RATE_LIMITED tag) thay vì buffer delay. Token bucket drop là safer than delay cho security events.
Đây là câu hỏi test bạn có thực sự hiểu sâu về trade-off của từng decision hay không. "What would break if you add X?" là kiểu câu hỏi senior interview.
🔍 Bonus — OpenSearch Config & Sizing 4 câu▼
D01 Walk me through shard sizing math: tại sao bạn chọn 5 primary shards cho 500M events/day? Nếu tenant tăng lên 10 tenants mỗi tenant 500M events, bạn scale như thế nào?
Trung bìnhOpenSearch+
▸ Đáp án gợi ý
// SHARD SIZING MATH: 500M EVENTS/DAY → 5 SHARDS
Rollover strategy: Index rollover khi size ≥ 50GB OR age ≥ 1 day. 5 shards chia đều trên hot data nodes. Với 1 replica, tổng = 10 shards per day index — nằm trong "less than 20 shards per GB heap" guideline của OpenSearch.
D02 Tại sao bạn dùng dynamic: strict và unmapped với enabled: false? Trade-off là gì với analyst queries?
Trung bìnhOpenSearch+
▸ Đáp án gợi ý
// dynamic:strict + unmapped:disabled — MAPPING EXPLOSION PREVENTION
Trade-off: unmapped với enabled: false (raw stored, not indexed): analysts CANNOT do full-text search or aggregation on unmapped.*. They can only retrieve exact stored values. Acceptable because unmapped = source-specific fields without OCSF equivalents — analysts primarily query OCSF standard fields.

Alternative: flattened type cho unmapped — allows keyword-style exact match on subkeys without mapping explosion. Better for analyst use-cases.
D03 Tại sao bạn set refresh_interval: 30s thay vì default 1s? Với SOC analyst context, đây có phải trade-off hợp lý không?
DễOpenSearch+
▸ Đáp án gợi ý
// REFRESH INTERVAL: INDEXING THROUGHPUT vs SEARCH FRESHNESS
Justification: SOC analysts investigates incidents that happened minutes or hours ago — họ không cần data visible trong 1 giây. 30 giây delay hoàn toàn acceptable. Benefit: ~30% indexing throughput improvement và ít GC pressure hơn từ segment merging.

Exception: Nếu có SIEM use-case cần real-time alerting (< 5s), có thể set 1-5s cho alert-specific indices, nhưng giữ 30s cho historical analytics indices.
D04 ISM policy: tại sao force-merge về 1 segment khi transition sang warm tier? Làm sao force-merge ảnh hưởng đến OpenSearch cluster performance trong quá trình chạy?
KhóOpenSearch+
▸ Đáp án gợi ý
// ISM: WARM TIER FORCE-MERGE — BENEFIT vs PERFORMANCE IMPACT
Why force-merge: Hot indices accumulate many small segments from continuous writes. Querying requires scanning all segments → slow for 7-day incident investigations. Force-merge consolidates to 1 segment → read path is dramatically faster.

Performance impact: Force-merge is extremely I/O and CPU intensive. Best practice: trigger in ISM policy with time-of-day condition (02:00-04:00 local time), and never force-merge on an actively-written index. That's why we wait for warm transition (index is read-only before force-merge).
