Q2 Làm thế nào để đạt uptime ≥ 99.5% — Circuit Breaker, Blue-Green, Multi-AZ?
Architecture
+
📌 KPI 1: System Uptime ≥ 99.5%
  • Multi-AZ Deployment: AWS RDS Multi-AZ, EKS nodes spread across ≥ 3 AZ. Không có single point of failure.
  • HPA + PDB: Kubernetes HorizontalPodAutoscaler scale theo custom metrics (active device connections). PodDisruptionBudget đảm bảo ≥ 2 replicas luôn running khi rolling update.
  • Circuit Breaker: Golang service dùng sony/gobreaker — nếu upstream fail > threshold, stop gọi ngay, tránh cascade failure.
  • Blue-Green Deploy: traffic switch qua ALB weighted routing. Rollback trong 30 giây nếu error rate tăng sau deploy.
  • Runbook + On-Call: Datadog alert → PagerDuty → on-call engineer. Mỗi loại incident có runbook chuẩn bị sẵn.
  • RTO/RPO: RTO < 15 phút, RPO < 1 giờ. PostgreSQL Point-in-Time Recovery, S3 Cross-Region Replication.

Q4 Policy Enforcement Engine — từ Admin tạo policy đến device apply?
MDM Protocol
+
📌 JD: "policy enforcement, remote commands"

Dùng mô hình Declarative + Event-Driven. Admin định nghĩa "desired state" — Engine đảm bảo device luôn đạt state đó.

1
Admin tạo policy trên Console: JSON Schema với password, apps, network, screen rules → POST /api/policies
2
Policy Store: lưu PostgreSQL JSONB, version-controlled. Publish event policy.updated lên Kafka.
3
Policy Dispatcher consume event → query affected devices (by org_id / group / tag) → batch silent push APNs/FCM.
4
Device Agent nhận silent push → GET /mdm/policies/{version} → apply locally: password, app whitelist, network config.
5
Compliance Report: Agent gửi POST /devices/{id}/compliance → server cập nhật compliance_status. Non-compliant → auto-remediate hoặc alert.

Q5 API response time < 200ms — đạt được bằng cách nào?
Performance
+
📌 KPI 2: Average API Response Time < 200ms
  • Redis Caching: device metadata, policy data cache với TTL phù hợp. Cache-aside pattern. Hit rate mục tiêu >85%.
  • DB Optimization: Index đúng columns (device_id, org_id, policy_id). Connection pooling với PgBouncer. Query explain analyze định kỳ.
  • Async Processing: Bulk commands, reporting dùng Kafka queue. API trả 202 Accepted ngay, worker xử lý background.
  • GraphQL DataLoader: Batch và cache queries, tránh N+1 problem trong Admin Console.
  • CDN: CloudFront cho Admin Console static assets, S3 pre-signed URLs cho file downloads.
  • Profiling: Datadog APM trace mọi request. p95 latency alert nếu vượt 500ms. Golang pprof cho hot path optimization.

Q10 Quy trình Code Review — đảm bảo quality, security, performance?
Leadership
+
📌 JD: "Code review, đảm bảo chất lượng code, bảo mật và performance" + Toolstack: GitHub Team, SonarCloud

Q18 APNs và FCM hoạt động như thế nào trong MDM context?
Protocol
+
  • APNs: MDM server gửi silent push (content-available: 1) qua HTTP/2. Device nhận → check-in MDM server → pull pending commands. Apple yêu cầu APNs certificate (renew yearly!). Connection dùng TLS client certificate.
  • FCM: MDM server gửi data-only message (không phải display notification). Android agent nhận → pull commands từ MDM server. Hỗ trợ cả foreground và background delivery.
  • Key principle: Push là "knock on door" — payload thực tế được pull từ MDM server qua mTLS. Không gửi sensitive data qua push channel (security reason).
  • Failure handling: Token rotation, delivery failure retry với exponential backoff, Dead Letter Queue cho devices offline quá lâu.
Q19 Remote Wipe an toàn — Double Authorization, Audit Log?
Security
+
  • Remote Wipe là lệnh nhạy cảm nhất — cần double authorization: 2 admin confirm trong UI, mỗi người dùng separate session.
  • Audit log đầy đủ: who triggered, when, device info, IP address, admin session ID. Immutable log, retain 1 năm.
  • Gửi lệnh qua encrypted channel (mTLS) — không thể intercept và replay.
  • Android: DevicePolicyManager.wipeData() → factory reset. iOS: EraseDevice MDM command.
  • Selective Wipe cho BYOD: chỉ xóa Work Profile / managed apps. Personal data không bị touch.
  • Confirm wipe thành công qua device last check-in trước khi device offline hoàn toàn.
Q20 Certificate Management cho 10,000+ devices fleet?
Security
+
  • SCEP Protocol: Dùng SCEP (Simple Certificate Enrollment Protocol) cho enrollment. Hoặc tích hợp HashiCorp Vault PKI / AWS Certificate Manager.
  • Unique per-device cert: CN = device_id. Không share certificate giữa devices. Compromise 1 device không ảnh hưởng device khác.
  • Auto-renewal 30 ngày trước expiry: Device tự request certificate mới khi còn 30 ngày. Silent, không interrupt user.
  • Revocation: CRL (Certificate Revocation List) hoặc OCSP. Revoke ngay khi device lost/stolen/unenrolled. Revoked cert → device mất quyền pull commands.
  • Certificate metadata: Lưu PostgreSQL (expiry, device mapping, revocation status), cache Redis cho fast lookup.
  • Alert: Automated alert 60 ngày trước batch expiry. Dashboard hiển thị cert health per device.

Q23 Scale MDM từ 1,000 lên 100,000 devices — Kubernetes HPA, Kafka, Sharding?
Scalability
+
  • Horizontal Scaling: Kubernetes HPA dựa trên custom metrics (active device connections, command queue depth), không chỉ CPU.
  • Database: Read replicas cho query-heavy workload. Partition bảng device_commands theo time (PostgreSQL partitioning). Connection pooling PgBouncer.
  • Push Architecture: Device không maintain persistent TCP connection — dùng push + pull model. 100k devices x silent push = 100k HTTP/2 requests → Kafka buffer để smooth burst.
  • Sharding: Partition devices theo organization_id. Mỗi shard có dedicated DB schema.
  • Caching: Policy data aggressive cache (ít thay đổi, read nhiều). Redis Cluster cho cache layer.
  • Load Test: k6 / Locust simulate 100k device check-ins trước khi go-live. Identify bottleneck trên staging.
Q24 Monitoring và Alerting — Datadog, Sentry, SLI/SLO setup?
Monitoring
+
📌 Toolstack: Datadog + Sentry — Monitoring và error tracking
  • Metrics: Datadog APM + custom metrics: devices_online_count, command_latency_p99, enrollment_rate, policy_compliance_rate.
  • Logs: Structured JSON logs → Datadog Log Management → saved search cho common issues (enrollment failure, push failure).
  • Traces: Distributed tracing cho cross-service requests. Identify bottleneck trong enrollment flow.
  • Alert Tiers: P0 (page immediately, wake on-call), P1 (page in 15 min), P2 (Jira ticket). Không alert fatigue — tune thresholds carefully.
  • SLI/SLO: Uptime 99.5%, API p95 < 200ms, enrollment success rate > 99%. Error budget tracking.
  • Dashboards: Executive (uptime, devices enrolled, compliance rate) + Engineering (latency heatmap, error rate, queue depth).
Q25 Docker + Kubernetes (EKS) — dùng thế nào trong MDM project?
Infrastructure
+
📌 JD: "Cloud: AWS (EC2, S3, RDS, Lambda, IoT Core), Docker, Kubernetes"
  • Docker: Multi-stage build minimize image size. Non-root user (security). Distroless base image cho production. Trivy scan trong CI.
  • Deployment: Stateless services (API, worker) dùng Deployment. Stateful (Kafka, Redis) dùng StatefulSet.

Q26 Database Migration không downtime — Flyway, backward-compatible pattern?
Database
+

  • Backward-compatible pattern (không bao giờ rename/drop trong 1 deployment):
1
Add column nullable (không breaking change) — deploy migration
2

3

4
Deploy code chỉ read cột mới
5

Security audit, performance bottleneck analysis, test coverage gap. Prioritize backlog theo impact/effort. Propose improvement roadmap cho CTO. Quick wins list (high impact, low effort).

60–90 Ngày — Execute & Build Trust

Execute quick wins để build credibility. Bắt đầu refactor module critical nhất. Thiết lập coding standards + review process nếu chưa có. Team sees results, not just plans.

Nguyên tắc xuyên suốt

Không phán xét code cũ — đó là technical context. Mỗi quyết định đều có lý do tại thời điểm đó. Build on existing strengths, không phủ nhận toàn bộ.

Q29 Đảm bảo sản phẩm đạt chất lượng tương đương Scalefusion / MobiStyle?
Strategy
+
📌 JD: "đảm bảo sản phẩm MDM đạt chất lượng tương đương Scalefusion, MobiStyle"
  • Competitive Analysis: Trial Scalefusion + MobiStyle. Document feature matrix: tính năng nào họ có, performance benchmark (enrollment time, command latency, policy apply time).
  • Core First: Focus vào reliability của core: enrollment success rate >99%, command execution <5s, policy apply trong 1 phút. Đây là foundation — fancy features sau.
  • Admin UX: IT admin là primary user. Ưu tiên: bulk operations (select 1000 devices → apply policy), fast search, clear compliance dashboard.
  • Dogfooding: Dùng chính product để manage internal devices từ ngày đầu. Detect issues trước khi customer thấy.
  • Customer feedback loop sớm: Beta program với 3-5 customers từ tháng 2. Real usage = real feedback. Không build trong bóng tối.
  • Không copy blindly: Understand why Scalefusion làm vậy. Đôi khi họ làm vì legacy constraints — ta có thể làm better.
Q30 Câu hỏi ngược lại cho interviewer — bạn sẽ hỏi gì?
Closing
+

Hỏi ngược lại ít nhất 2-3 câu — cho thấy bạn đang evaluate họ, không chỉ xin việc. Đây là dấu hiệu của Tech Lead tư duy chủ động.

1
"MDM platform hiện tại đang ở giai đoạn nào?" — Greenfield hay có legacy codebase? Số devices đang manage? Đang dùng vendor nào hay tự build từ đầu?
2
"Team engineering hiện tại thế nào?" — Bao nhiêu người, seniority level, skill set? Sub Lead đã có chưa? Remote hay on-site?
3
"Technical challenge lớn nhất hiện tại là gì?" — Câu hỏi này reveal rất nhiều về state thực tế của team và product.
4
"Roadmap 12 tháng — tính năng nào được ưu tiên nhất?" — Hiểu business direction để biết tech investment nên focus vào đâu.
5
"Tech Lead có quyền quyết định tech stack hay cần approval từ CTO?" — Understand autonomy level. Nếu mọi quyết định kỹ thuật đều cần CTO approve, đó là red flag.

Interview Tips & Strategy

STAR Method · Quantify · Ownership Mindset

🎯 STAR Method

Situation → Task → Action → Result. Mọi câu behavioral phải có cấu trúc này. Result phải được quantify.

📊 Always Quantify

"Giảm latency 40%", "onboard 3 junior devs", "0 critical bugs trong 6 tháng". Số cụ thể tạo credibility.

🔑 Ownership Mindset

Không đổ lỗi cho constraints, team cũ, hay PM. Thể hiện bạn làm chủ vấn đề và tìm cách giải quyết trong context đó.

❓ Không biết → Approach

"Tôi chưa làm X trực tiếp, nhưng đây là cách tôi tiếp cận..." + research steps + timeline. Tốt hơn im lặng hoặc bluff.
