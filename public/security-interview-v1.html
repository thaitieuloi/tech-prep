N01 DDoS attack là gì? Các loại DDoS và cách phòng chống ở tầng network + application? CRITICAL ›

DDoS (Distributed Denial of Service) = hacker dùng botnet (hàng nghìn máy bị compromise) gửi flood requests đến target, làm hệ thống quá tải và không phục vụ được user thật.

Volume Attack (L3/L4)
UDP flood, ICMP flood, SYN flood — làm nghẽn bandwidth/pipe. Đo bằng Gbps.
Protocol Attack (L4)
SYN flood, Smurf, Ping of Death — khai thác TCP handshake, exhausts server resources.
Application Attack (L7)
HTTP flood, Slowloris, API abuse — khó phân biệt với traffic thật. Đo bằng RPS.

Defense layers:

  • 1
    CDN + WAF (Cloudflare/AWS Shield): Absorb volume attacks, anycast routing phân tán traffic toàn cầu. AWS Shield Standard free, Advanced cho enterprise.
  • 2
    Rate Limiting tại API Gateway: Giới hạn requests/second per IP, per user, per endpoint. VD: 100 req/s per IP, 10 req/s cho login endpoint.
  • 3
    SYN Cookies (OS level): Chống SYN flood — server không allocate state cho connection cho đến khi handshake hoàn tất.
  • 4
    Connection timeout + max connections: Config tại load balancer, reject connections vượt threshold.
  • 5
    Auto-scaling: Tăng capacity khi load tăng đột biến, sau đó scale down và block offending IPs.
spring boot// Rate limiting với Bucket4j + Redis
@Component
public class RateLimitFilter implements Filter {
    private final Map<String, Bucket> cache = new ConcurrentHashMap<>();

    @Override
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) {
        String ip = ((HttpServletRequest) req).getRemoteAddr();
        Bucket bucket = cache.computeIfAbsent(ip, k ->
            Bucket.builder()
                .addLimit(Bandwidth.classic(100, Refill.intervally(100, Duration.ofMinutes(1))))
                .build()
        );
        if (bucket.tryConsume(1)) {
            chain.doFilter(req, res);
        } else {
            ((HttpServletResponse) res).setStatus(429); // Too Many Requests
            return;
        }
    }
}
Trong phỏng vấn: "For L7 DDoS, we implement rate limiting at the API Gateway level and use anomaly detection — if an IP sends 1000 requests in 10 seconds for the login endpoint, it's automatically blocked and flagged."
N02 MITM (Man-in-the-Middle) attack là gì? TLS/HTTPS bảo vệ như thế nào? HSTS là gì? CRITICAL ›

MITM = attacker đặt mình giữa client và server, nghe lén hoặc modify traffic mà cả hai đầu không biết.

MITM Attack vs TLS Defense

↓ Attacker intercepts ↓

↓ Forward to Server ↓
Server ← response
vs
✅ With TLS 1.3

↓ Verify certificate ↓
Encrypted channel
↓ Attacker sees: garbage ↓

  • 1
    TLS 1.3: Enforce HTTPS, disable TLS 1.0/1.1, disable weak ciphers (RC4, 3DES, MD5-based)
  • 2
    HSTS (HTTP Strict Transport Security): Browser header buộc dùng HTTPS ngay cả khi user gõ HTTP. Chống SSL Stripping attack.
  • 3
    Certificate Pinning: Mobile app/service chỉ trust certificate cụ thể, không trust CA chain chung. Chống rogue certificate.
  • 4
    mTLS (Mutual TLS): Trong microservices — cả client VÀ server đều verify certificate nhau. Zero Trust networking.

A01 IDOR (Insecure Direct Object Reference) — Attack, demo code vulnerable vs secure, và cách test. CRITICAL ›

IDOR = Attacker thay đổi ID trong URL/request để truy cập object thuộc về user khác. Là dạng Broken Access Control phổ biến nhất.

Attack: User A có order ID=100. Họ thay URL thành /api/orders/101 → Lấy được order của User B vì server chỉ check authentication (đăng nhập chưa) mà không check authorization (có quyền với resource này không).
java — VULNERABLE// ❌ IDOR vulnerable — chỉ check auth token, không check ownership
@GetMapping("/api/orders/{orderId}")
public Order getOrder(@PathVariable Long orderId) {
    // Anyone logged in can access ANY order!
    return orderRepo.findById(orderId)
        .orElseThrow(() -> new NotFoundException("Order not found"));
}
java — SECURE// ✅ SECURE — Check ownership before returning data
@GetMapping("/api/orders/{orderId}")
public Order getOrder(@PathVariable Long orderId,
                      @AuthenticationPrincipal UserDetails currentUser) {
    Order order = orderRepo.findById(orderId)
        .orElseThrow(() -> new NotFoundException("Order not found"));

    // CRITICAL: verify ownership
    if (!order.getUserId().equals(currentUser.getId())) {
        // Return 404, not 403 — don't reveal resource exists
        throw new NotFoundException("Order not found");
    }
    return order;
}

// ✅ Better: query already scoped to current user
@Query("SELECT o FROM Order o WHERE o.id = :id AND o.userId = :userId")
Optional<Order> findByIdAndUserId(@Param("id") Long id, @Param("userId") Long userId);
Return 404 thay vì 403 cho IDOR — không để attacker biết resource có tồn tại hay không (information leakage). Đây là "security through ambiguity" nhưng hữu ích.
A02 JWT Attacks — alg:none, algorithm confusion, weak secret brute force. Cách secure JWT properly. CRITICAL ›

JWT có 3 phần: header.payload.signature. Nhiều implementation có lỗi nghiêm trọng trong việc verify.

Attack 1: alg:none
Attacker đổi header thành "alg":"none", xóa signature → server không verify! Bug trong một số JWT libraries.
Attack 2: RS256 → HS256
Server dùng RS256 (public key). Attacker dùng public key như HMAC secret → forge valid token.
Attack 3: Weak Secret
Secret "password123" → attacker brute force offline bằng hashcat trong vài giây.
java — secure JWT// ✅ Secure JWT validation với explicit algorithm
@Component
public class JwtService {

    // ✅ Use strong secret (≥256 bits for HS256)
    @Value("${jwt.secret}")  // from secrets manager, NOT hardcoded
    private String secret;

    public Claims parseToken(String token) {
        return Jwts.parserBuilder()
            .setSigningKey(Keys.hmacShaKeyFor(secret.getBytes()))
            // ✅ Explicit algorithm — reject "none" and algorithm confusion
            .requireAlgorithm("HS256")
            .build()
            .parseClaimsJws(token)  // throws if invalid/expired
            .getBody();
    }

    public String generateToken(User user) {
        return Jwts.builder()
            .setSubject(user.getId().toString())
            .setIssuedAt(new Date())
            .setExpiration(new Date(System.currentTimeMillis() + 900_000)) // 15 min
            .signWith(Keys.hmacShaKeyFor(secret.getBytes()), SignatureAlgorithm.HS256)
            .compact();
    }
}

Short expiry (15–30 min)Limit blast radius nếu token bị steal
Refresh token rotationPhát hiện token theft (old refresh token dùng = alert)
Secret ≥256 bitsKhông brute-forceable
Store in httpOnly cookieChống XSS lấy token từ localStorage
Validate all claims (iss, aud, exp)Chống token reuse từ service khác

java — spring security// ✅ CSRF protection với Spring Security
@Configuration
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // For stateful web apps: enable CSRF with token
            .csrf(csrf -> csrf
                .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
                .ignoringRequestMatchers("/api/webhook/**")  // external callbacks
            )
            // For stateless REST APIs: disable CSRF (use JWT instead)

            .sessionManagement(s -> s
                .sessionCreationPolicy(SessionCreationPolicy.STATELESS));
        return http.build();

// Cookie settings — SameSite protection
@Bean
public CookieSameSiteSupplier applicationCookieSameSiteSupplier() {
    return CookieSameSiteSupplier.ofStrict(); // SameSite=Strict
}
SameSite=Strict
Cookie KHÔNG được gửi trong cross-site requests. Strongest protection, nhưng có thể break UX khi click link từ email.
SameSite=Lax
Cookie gửi khi navigate (GET), không gửi khi POST/PUT/DELETE cross-site. Good balance giữa security và UX.
A04 SSRF (Server-Side Request Forgery) — Attack vector, impact, và defense trong Java microservices. CRITICAL ›

SSRF = Server bị trick để gửi request đến internal resources hoặc external services. Nguy hiểm đặc biệt trên cloud — attacker có thể reach 169.254.169.254 (AWS metadata endpoint) để lấy IAM credentials.

AWS SSRF attack: curl http://169.254.169.254/latest/meta-data/iam/security-credentials/ → Lấy được temporary AWS credentials → Access S3, RDS, mọi service trong AWS account!
java — VULNERABLE// ❌ SSRF vulnerable — fetch URL từ user input
@PostMapping("/fetch-url")
public String fetchUrl(@RequestParam String url) {
    // Attack: url = "http://169.254.169.254/latest/meta-data/iam/..."
    // Attack: url = "file:///etc/passwd"
    // Attack: url = "http://internal-service:8080/admin"
    return restTemplate.getForObject(url, String.class); // DANGEROUS!
}
java — SECURE// ✅ SSRF defense — validate and allowlist URLs
@Service
public class SafeHttpClient {
    // Allowlist of permitted domains
    private static final Set<String> ALLOWED_DOMAINS = Set.of(
        "api.trusted-partner.com",
        "cdn.example.com"
    );

    public String safeFetch(String urlStr) {
        try {
            URL url = new URL(urlStr);

            // 1. Only allow https
            if (!"https".equals(url.getProtocol()))
                throw new IllegalArgumentException("Only HTTPS allowed");

            // 2. Allowlist domains
            if (!ALLOWED_DOMAINS.contains(url.getHost()))
                throw new SecurityException("Domain not allowed: " + url.getHost());

            // 3. Resolve IP and block private ranges
            InetAddress addr = InetAddress.getByName(url.getHost());
            if (addr.isLoopbackAddress() || addr.isSiteLocalAddress()
                || addr.isLinkLocalAddress())
                throw new SecurityException("Private IP ranges not allowed");

            // 4. No redirect following
            return restTemplate.getForObject(urlStr, String.class);
        } catch (Exception e) {
            throw new SecurityException("URL validation failed", e);
        }
    }
}
A05 Insecure Deserialization — Java deserialization attacks, gadget chains, và cách phòng chống. CRITICAL ›

Insecure Deserialization = Khi app deserialize untrusted data (từ network, cookie, file) mà không validate, attacker có thể craft malicious serialized object để execute arbitrary code (RCE).

Java Gadget Chains: Apache Commons Collections, Spring Framework, Hibernate — nhiều Java libraries có "gadget chains" mà khi deserialized theo thứ tự nhất định, trigger Runtime.exec() để chạy OS commands.

java — VULNERABLE// ❌ NEVER deserialize untrusted data with Java native serialization
byte[] data = request.getBody();
ObjectInputStream ois = new ObjectInputStream(new ByteArrayInputStream(data));
Object obj = ois.readObject();  // RCE possible if malicious gadget chain!
java — SECURE// ✅ Use JSON/Protobuf instead of Java serialization
// Never use ObjectInputStream for untrusted data

// If you MUST use Java serialization, use ValidatingObjectInputStream
import org.apache.commons.io.serialization.ValidatingObjectInputStream;

ValidatingObjectInputStream vois = new ValidatingObjectInputStream(inputStream);
// Whitelist only known safe classes
vois.accept(MyExpectedClass.class, OtherSafeClass.class);
// Reject everything else — throws exception for unknown classes
MyExpectedClass obj = (MyExpectedClass) vois.readObject();

// ✅ Better: use Jackson JSON (no gadget chains)
ObjectMapper mapper = new ObjectMapper();
mapper.activateDefaultTyping(
    LaissezFaireSubTypeValidator.instance,
    ObjectMapper.DefaultTyping.NON_FINAL  // restrict polymorphism
);
MyDto dto = mapper.readValue(jsonString, MyDto.class);
Real-world: Log4Shell (CVE-2021-44228) exploit JNDI lookup → remote classloading → deserialization gadget chain → RCE. Đây là lý do tại sao upgrading log4j là critical.
A06 SQL Injection — demo vulnerable vs secure code trong Java/Spring. Các dạng SQLi và cách test. CRITICAL ›

SQL Injection = user input được concatenate trực tiếp vào SQL query. Attacker inject SQL code để bypass auth, dump database, hoặc drop tables.

Classic SQLi
Input: ' OR '1'='1 → Returns all rows. Bypass login.
Blind SQLi
No output shown. Attacker infers data via true/false hoặc time delays (SLEEP(5)).
Out-of-band SQLi
Exfiltrate data via DNS/HTTP từ DB server. Khó detect hơn.
java — VULNERABLE vs SECURE// ❌ String concatenation — NEVER
public User login(String username, String password) {
    String sql = "SELECT * FROM users WHERE username='"
        + username + "' AND password='" + password + "'";
    // Attack: username = "admin'--"  → bypasses password check!
    return jdbcTemplate.queryForObject(sql, User.class);
}

// ✅ Parameterized query
public User login(String username, String password) {
    String sql = "SELECT * FROM users WHERE username=? AND password_hash=?";
    return jdbcTemplate.queryForObject(sql,
        new Object[]{username, hashPassword(password)}, User.class);
}

// ✅ JPA Named Query (auto-safe, best practice)
@Query("SELECT u FROM User u WHERE u.username = :username")
Optional<User> findByUsername(@Param("username") String username);

// ✅ Spring Data method naming (safest — no SQL at all)
Optional<User> findByUsernameAndEnabled(String username, Boolean enabled);
Rule: NEVER concatenate user input vào SQL/LDAP/OS commands. Snyk Code (SAST) sẽ flag taint flows từ HTTP input → SQL execution trong code review. Test bằng input: ' OR 1=1--, ; DROP TABLE--, UNION SELECT--

A08 Spring Security — SecurityFilterChain, deny-by-default, RBAC với @PreAuthorize, method-level security. HIGH ›
Authentication (AuthN) — "Who are you?"
Verify identity: username+password, JWT, OAuth. Kết quả: JWT issued.
Authorization (AuthZ) — "What can you do?"
Check permissions sau authn. RBAC/ABAC. Kết quả: allow hoặc deny resource access.
java — spring security 6@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        return http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .requestMatchers(HttpMethod.DELETE, "/api/**").hasRole("MANAGER")
                .anyRequest().authenticated())   // deny-by-default!
            .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}

// Method-level fine-grained authorization
@GetMapping("/orders/{id}")
@PreAuthorize("hasRole('USER') and @orderSecurity.isOwner(#id, authentication)")
public Order getOrder(@PathVariable Long id) { ... }

@DeleteMapping("/orders/{id}")
@PreAuthorize("hasRole('ADMIN')")
public void deleteOrder(@PathVariable Long id) { ... }
Key: anyRequest().authenticated() là last rule → deny by default. Bất kỳ endpoint mới nào chưa có rule sẽ require authentication tự động — prevent accidentally public endpoints.

D01 Encryption at Rest vs Encryption in Transit — khi nào cần cái nào? AES-256, bcrypt, argon2. CRITICAL ›
Encryption in Transit
Bảo vệ: Data đang di chuyển trên network
Tool: TLS 1.3, HTTPS, mTLS
Chống: MITM, eavesdropping, packet sniffing
Required: Mọi network communication
Encryption at Rest
Bảo vệ: Data stored trên disk/DB
Tool: AES-256-GCM, AWS KMS, disk encryption
Chống: Physical theft, backup exposure, DB dump
Required: PII, credentials, payment data
java — encryption// ✅ Password hashing với BCrypt (adaptive, slow by design)
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12); // strength=12, ~250ms
}
// Usage: encoder.encode("plaintext") → "$2a$12$..."
// Verify: encoder.matches("plaintext", hash)

// ✅ Field-level encryption với AES-256-GCM
public class AesEncryption {
    private static final int KEY_SIZE = 256;
    private static final int IV_SIZE = 12; // GCM nonce
    private static final int TAG_SIZE = 128; // auth tag bits

    public byte[] encrypt(SecretKey key, byte[] plaintext) throws Exception {
        byte[] iv = new byte[IV_SIZE];
        new SecureRandom().nextBytes(iv); // random nonce each time!
        Cipher cipher = Cipher.getInstance("AES/GCM/NoPadding");
        cipher.init(Cipher.ENCRYPT_MODE, key, new GCMParameterSpec(TAG_SIZE, iv));
        byte[] encrypted = cipher.doFinal(plaintext);
        // Prepend IV for decryption
        return ByteBuffer.allocate(iv.length + encrypted.length)
            .put(iv).put(encrypted).array();
    }
}

D02 Secrets Management — tại sao không dùng env vars đơn giản? Vault, AWS Secrets Manager, rotation. CRITICAL ›

Environment variables tốt hơn hardcoded secrets, nhưng vẫn có vấn đề: chúng visible trong process list, container inspect, và không có audit log.

Level 0 — Hardcoded (❌ Never)
String password = "mySecret"; — Trong git history mãi mãi. Instant compromise nếu repo leak.
Level 1 — .env file (⚠️ Only local)
.env + .gitignore. Còn visible trong container env, logs. Không có rotation.
Level 2 — CI/CD Secrets (✓ Good)
GitHub Secrets, GitLab CI variables — encrypted, không visible trong logs. Nhưng thiếu rotation và audit log.
Level 3 — Secrets Manager (✅ Production)
AWS Secrets Manager / HashiCorp Vault: encrypted, versioned, auto-rotation, audit log, fine-grained access control.
java — aws secrets manager// ✅ Spring Cloud AWS — auto-load secrets
spring:
  config:
    import: "aws-secretsmanager:/myapp/prod/"
  datasource:
    url: ${DB_URL}       # auto-populated from Secrets Manager
    password: ${DB_PASS} # rotated automatically every 30 days

// ✅ Programmatic access with auto-rotation
@Service
public class SecretService {
    private final SecretsManagerClient client;

    public String getSecret(String secretName) {
        GetSecretValueRequest req = GetSecretValueRequest.builder()
            .secretId(secretName)
            .build();
        // Always fetches latest version — auto-rotation transparent
        return client.getSecretValue(req).secretString();
    }
}
Auto-rotation strategy: AWS Secrets Manager rotates DB password → updates secret → your app fetches new value on next request → no downtime, no manual intervention.
D03 Data masking và PII protection — log sanitization, sensitive data in API responses, GDPR compliance. HIGH ›

Data masking = che giấu sensitive data (PII) trong logs, API responses, error messages, và non-production environments.

java — log masking// ❌ Sensitive data in logs — PII/GDPR violation
log.info("Processing payment for user={}, card={}, amount={}",
    user.getEmail(), user.getCreditCard(), amount); // NEVER!

// ✅ Log masking với custom converter
@Converter
public class SensitiveDataMasker {
    public static String maskEmail(String email) {
        if (email == null) return null;
        int at = email.indexOf('@');
        if (at <= 1) return "***";
        return email.charAt(0) + "***" + email.substring(at); // j***@gmail.com
    }
    public static String maskCard(String card) {
        return "****-****-****-" + card.substring(card.length() - 4);
    }
}
log.info("Processing payment for user={}, card={}",
    SensitiveDataMasker.maskEmail(email),
    SensitiveDataMasker.maskCard(card));
java — api response// ✅ Never expose sensitive fields in API response
public class UserResponse {
    private Long id;
    private String username;
    @JsonIgnore  // never serialize
    private String password;
    @JsonProperty(access = JsonProperty.Access.WRITE_ONLY)
    private String ssn;  // accept in request, never in response
    @JsonSerialize(using = MaskedEmailSerializer.class)
    private String email; // j***@gmail.com in response
}
GDPR principle of data minimization: chỉ collect, store, và return data thực sự cần thiết. API không bao giờ return password hash, internal IDs unnecessary, hay PII không cần thiết cho frontend.

C01 Container escape là gì? Cách hacker thoát khỏi container lên host? Defense strategies. CRITICAL ›

Container escape = Attacker đã compromise được process trong container, sau đó exploit vulnerabilities để "thoát" ra ngoài container sandbox và access host OS hoặc các container khác.

Escape Vectors
• Running as root in container
• --privileged flag
• Mounted /var/run/docker.sock
• Kernel vulnerabilities (runc CVEs)
• Writable /proc or host filesystem mount
Defense
• Non-root user (UID >1000)
• Read-only root filesystem
• Drop ALL capabilities, add only needed
• No --privileged
• Seccomp + AppArmor profiles
• Never mount docker.sock
dockerfile + compose# ✅ Dockerfile — non-root, minimal
FROM amazoncorretto:21-alpine
RUN addgroup -S app && adduser -S app -G app
COPY --chown=app:app app.jar /app/app.jar
USER app
# Read-only filesystem (set at runtime)

# ✅ docker-compose security options
services:
  api:
    image: my-app:1.2.3
    read_only: true            # read-only root filesystem
    security_opt:
      - no-new-privileges:true # prevent privilege escalation
      - seccomp:./seccomp.json # restrict syscalls
    cap_drop:
      - ALL                    # drop all Linux capabilities
    cap_add:
      - NET_BIND_SERVICE       # only add what's needed
    tmpfs:
      - /tmp                   # writable tmp in memory only
    user: "1001:1001"
Docker socket (/var/run/docker.sock) mounted vào container = container có thể control Docker daemon = instant host compromise. Đây là common misconfiguration.
C02 Secrets in Docker images — làm sao hacker extract secrets từ image layers? Và cách prevent. CRITICAL ›

Docker image = stack of layers. Ngay cả khi bạn RUN rm secret.txt, layer trước đó vẫn còn trong image history — attacker có thể extract bằng docker history hoặc docker save + tar.

shell — attack# How attacker extracts secrets from image layers
docker history my-app:latest --no-trunc
# Shows: RUN echo "API_KEY=sk-abc123..." — still visible!

docker save my-app:latest | tar xv
# Extract each layer, find secrets in any layer
dockerfile — WRONG vs RIGHT// ❌ WRONG — secret in build arg, visible in history
ARG NPM_TOKEN
RUN echo "//registry.npmjs.org/:_authToken=${NPM_TOKEN}" > .npmrc
RUN npm install
RUN rm .npmrc  // TOO LATE — still in previous layer!

// ✅ RIGHT — use Docker BuildKit secrets (not in layers)
// Dockerfile:
RUN --mount=type=secret,id=npmtoken \
    NPM_TOKEN=$(cat /run/secrets/npmtoken) \
    npm install
// Build command:
// docker build --secret id=npmtoken,env=NPM_TOKEN .
// Secret is NEVER written to any layer!
Rule: Không có secret nào trong ENV, ARG, hay RUN commands. Dùng BuildKit secrets cho build-time secrets. Runtime secrets qua environment variables hoặc Secrets Manager.

S03 CVE Lifecycle — từ discovery đến patch. Zero-day là gì? Cách respond khi Zero-day được công bố. HIGH ›
Discovery (Day 0)
Security researcher hoặc attacker tìm ra vulnerability. "Zero-day" = chưa có patch nào tồn tại khi được exploit.
CVE Assignment
MITRE gán CVE ID (CVE-YYYY-NNNNN). NVD publish với CVSS score. Snyk DB update trong vài giờ.
Public Disclosure + PoC
Proof-of-Concept exploit code được publish. Window giữa PoC và patch = critical danger zone.
Vendor Patch
Library maintainer release fix. Downstream consumers (bạn) phải update trong SLA dựa trên severity.
Remediation Deployed
Your team upgrades, tests, deploys. Snyk re-scan xác nhận clean. Vulnerability closed.

Quy trình respond Zero-day (ví dụ Log4Shell):

  • 1
    Immediate (Hour 1): Snyk scan toàn bộ repo — identify affected services. Dùng SBOM nếu có.
  • 2
    Workaround (Hour 2): Apply mitigation (Log4Shell: set log4j2.formatMsgNoLookups=true) nếu upgrade chưa kịp test.
  • 3
    Fix (Hours 4-24): Upgrade đến safe version, test, deploy staging.
  • 4
    Verify (After deploy): Snyk re-scan, confirm không còn CVE. WAF rules update.
  • 5
    Post-mortem: Tại sao mất X giờ? SBOM có giúp không? Cải thiện process gì?

• No fix available

❌ Never Ignore

P02 CI/CD pipeline security — least privilege builds, secrets in pipeline, artifact signing với cosign. HIGH ›

CI/CD pipeline bản thân là attack surface — nếu hacker compromise pipeline, họ có thể inject malicious code vào artifacts được deploy lên production.

yaml — secure pipeline# ✅ Least privilege GitHub Actions
permissions:
  contents: read        # only read repo
  packages: write       # only push to registry
  security-events: write # upload SARIF only
  id-token: write       # for OIDC token (no stored secrets!)

# ✅ Pin action versions to commit SHA (not tag)
# tags can be moved; SHA is immutable
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2

# ✅ OIDC — no long-lived AWS credentials stored in GitHub
- name: Configure AWS Credentials (OIDC)
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::123456789:role/github-actions-deploy
    aws-region: ap-southeast-1
  # GitHub gets temp credentials via OIDC, no SECRET_KEY stored!

# ✅ Sign Docker image with cosign (supply chain integrity)
- name: Sign image with cosign
  run: |
    cosign sign --yes \
      -a "git-sha=${{ github.sha }}" \
      -a "workflow=${{ github.workflow }}" \
      ${{ env.IMAGE_TAG }}

# ✅ Verify signature before deploy
- name: Verify image signature
  run: cosign verify --certificate-identity-regexp='.*' ${{ env.IMAGE_TAG }}
Không pin actions bằng tag (@v4) — tag có thể bị overwrite. Dùng commit SHA để đảm bảo action code không thay đổi sau khi bạn review.

java — spring security@Configuration
public class SecurityHeadersConfig {
    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http.headers(headers -> headers
            .contentSecurityPolicy(csp -> csp
                .policyDirectives("default-src 'self'; script-src 'self'; frame-ancestors 'none'"))
            .frameOptions(frame -> frame.deny())
            .xssProtection(xss -> xss.disable()) // deprecated, use CSP instead
            .permissionsPolicy(pp -> pp
                .policy("camera=(), microphone=(), geolocation=()"))
            .httpStrictTransportSecurity(hsts -> hsts
                .includeSubDomains(true)
                .maxAgeInSeconds(31536000)
                .preload(true))
        );
        return http.build();
    }
}
Test security headers tại securityheaders.com hoặc OWASP ZAP — đây là quick win vì chỉ cần config, không cần code change lớn.
P04 Maven dependency conflict — nearest-wins rule, cách phát hiện và resolve conflict để đảm bảo safe version. HIGH ›

Dependency conflict = 2+ dependencies yêu cầu các versions khác nhau của cùng một library. Maven dùng "nearest-wins" rule — version gần root dependency tree nhất được chọn.

Nearest-Wins — Security Problem
my-app
 ├─ dep-A
 │   └─ commons-text:1.9     (depth 2) ← Maven picks this (nearest)
 └─ dep-B
     └─ dep-C
         └─ commons-text:1.10.0  (depth 3) ← SAFE version, but loses!

→ Maven picks 1.9 (vulnerable!) because it's nearest to root.
→ Solution: force 1.10.0 via dependencyManagement
bash + xml — detect and fix# Detect conflicts
mvn dependency:tree -Dverbose | grep "omitted for conflict"
mvn dependency:tree -Dincludes=org.apache.commons:commons-text

# Fix: force safe version regardless of depth
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.apache.commons</groupId>
      <artifactId>commons-text</artifactId>
      <version>1.10.0</version>  <!-- force safe version -->
    </dependency>
  </dependencies>
</dependencyManagement>

# Verify after fix
mvn dependency:tree -Dincludes=org.apache.commons:commons-text
# Should show: commons-text:1.10.0:compile (no "omitted" message)
Spring Boot BOM (spring-boot-starter-parent) pre-defines safe versions cho 200+ dependencies — import BOM để có baseline security và giảm conflicts đáng kể.
