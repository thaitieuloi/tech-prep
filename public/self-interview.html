RAG — Additional Details
Chunking strategy · Qdrant vs alternatives · Evaluation metrics
RAG-D
Why Qdrant specifically? Why 500 tokens chunk size? How do you evaluate RAG quality?
Tại sao Qdrant? Tại sao 500 tokens? Đánh giá chất lượng RAG thế nào?
›
Why Qdrant (from CV)
Written in Rust → extremely fast. Rich metadata filtering (filter by date, type, user). Native hybrid search (vector + sparse). Self-hostable — matches Glohow's on-prem requirement. Good Java client for Spring Boot integration.
vs Pinecone: vendor lock-in + expensive at scale
500 tokens — why?
500 tokens ≈ 375 words = one complete idea. Too small (<200): fragmented, loses context. Too large (>1000): noisy, LLM distracted. 50-token overlap prevents losing context at boundaries. Never split tables/code (atomic units).
Semantic chunking considered but too slow
EVALUATERAG Triad metrics:
  • ①Context Relevance: retrieval có tìm đúng chunks không?
  • ②Groundedness: answer có xuất phát từ context không?
  • ③Answer Relevance: answer có đúng câu hỏi không?
Tooling: Ragas hoặc DeepEval cho automated LLM-as-judge scoring. Production: track negative feedback rate + p95 latency.

RAG Bottlenecks — Full Solution Stack
Interviewer hint: "vector score may not accurately retrieve · too many docs = AI drafting"
BOT
What are all the RAG limitations and how do you solve each one?
Tất cả giới hạn của RAG và giải pháp cho từng cái?
›
Problem 1 — Low Precision
Vector retrieves similar but wrong chunks. Fix: Hybrid Search (Qdrant + Elasticsearch BM25). +25% recall for names/IDs.
Trade-off: 2 systems to maintain
Problem 2 — Context Overflow "Drafting"
K=20, all score~0.65 → LLM confused. Fix: Re-rank → keep top 4 only. Prompt: "Say I don't know if absent."
Trade-off: 80ms extra latency
Problem 3 — Hallucinations
LLM generates false info not in context. Fix: Temperature=0.1 + force citation + groundedness check post-generation.
Trade-off: 1 extra LLM call for check
Problem 4 — Recency Bias
Vector favors recent docs over relevant older ones. Fix: Time-decay scoring: score = similarity × decay(age). Recent docs get small boost, not dominant.
Trade-off: custom scoring logic

DIAGRAM — MICROSERVICE BEST PRACTICE: Kong · Istio · Keycloak · ELK · Prometheus
Kong Gateway — Why?
Single entry point cho tất cả clients. Xử lý cross-cutting concerns (auth, rate limit, logging) tại edge — không để từng service tự xử lý. Plugin ecosystem: JWT validation với Keycloak, IP restriction, request transformation.
Trade-off: single point of failure → cần HA cluster Kong
Istio Service Mesh — Why?
East-west traffic (service-to-service) cần mTLS, circuit breaker, retry, và observability — không code vào application. Envoy sidecar inject tự động. Istio control plane quản lý policy tập trung.
Trade-off: ~15% latency overhead, ops complexity cao
Database per Service — Why?
Mỗi service own schema riêng — loose coupling, independent deploy. Saga pattern cho distributed transactions thay vì 2PC. Event sourcing qua Kafka cho audit trail.
Trade-off: join queries phức tạp hơn — cần CQRS

Keycloak — Auth & Identity
OAuth2 · OIDC · SSO · RBAC · Token flows · Tích hợp với Spring Boot
KC1
How does Keycloak integrate with your microservice architecture? Explain the token flow.
Keycloak tích hợp thế nào với microservice? Giải thích token flow.
›
ANSWERKeycloak là Identity Provider (IdP) — issue JWT Access Token và Refresh Token. Kong Gateway validate token tại edge. Services downstream nhận token đã verified — không cần call Keycloak mỗi request.
FLOW
  • ①User login → POST /realms/{'{realm}'}/protocol/openid-connect/token → Keycloak trả về Access Token (JWT, 15min) + Refresh Token (7 days)
  • ②Client gửi request kèm Authorization: Bearer {'{token}'} → Kong intercept
  • ③Kong validate JWT signature (public key từ Keycloak JWKS endpoint) — không cần gọi Keycloak
  • ④Kong forward request kèm decoded claims (userId, roles) đến service
  • ⑤Service dùng Spring Security + @PreAuthorize để enforce RBAC theo roles trong JWT
TRADE-OFFJWT stateless = không thể revoke ngay lập tức. Solution: short-lived access token (15min) + Redis blacklist cho logout trước hạn. Hoặc dùng Keycloak introspection endpoint (online validation) cho high-security endpoints.
KEYCLOAK TOKEN FLOW — Authorization Code + PKCE (Web App)
KC2
How do you implement RBAC with Keycloak in Spring Boot microservices?
Implement RBAC với Keycloak trong Spring Boot microservices thế nào?
›
SETUPKeycloak Realm → Clients (per service) → Roles (ADMIN, USER, READONLY) → Assign to Users/Groups. Spring Boot config: spring-boot-starter-oauth2-resource-server + JwtDecoder pointing to Keycloak JWKS URI.
CODERoles nằm trong JWT claims (realm_access.roles). Custom JwtAuthenticationConverter extract roles thành Spring Security Authorities. Dùng @PreAuthorize("hasRole('ADMIN')") ở method level.
GOTCHAKeycloak role prefix: Spring Security tự thêm ROLE_ prefix. Nếu Keycloak role là admin, annotation phải là hasRole('admin') không phải ADMIN. Hay dùng hasAuthority('ROLE_admin') cho chắc.

KC3
How do you handle token revocation and logout in a stateless JWT system?
Xử lý token revocation và logout trong hệ thống JWT stateless thế nào?
›
PROBLEMJWT stateless = không thể revoke. Nếu token bị đánh cắp, hacker dùng được đến khi hết hạn (15min). Logout không thực sự vô hiệu hóa được JWT.
SOLUTION 1Redis blacklist: Khi logout, lưu JTI (JWT ID) vào Redis với TTL = remaining lifetime. Kong check blacklist mỗi request tại plugin level. Trade-off: thêm Redis lookup cho mỗi request (~1ms).
SOLUTION 2Short TTL + Refresh rotation: Access token 5min, Refresh token single-use (rotate sau mỗi lần dùng). Compromise phát hiện khi Refresh Token bị dùng lần 2 — Keycloak revoke toàn bộ session.
MY CHOICEHybrid: 15min access token + Redis blacklist chỉ cho logout. Không check blacklist cho mọi request — chỉ check khi Keycloak session_state thay đổi.

What is a service mesh? Why use Istio instead of handling this in application code?
Service mesh là gì? Tại sao dùng Istio thay vì xử lý trong application code?

ANSWERService mesh là infrastructure layer xử lý service-to-service communication (east-west traffic) — mTLS, retry, circuit breaker, load balancing, observability — không cần code vào application. Istio inject Envoy sidecar proxy vào mỗi pod, transparent với app.
WHY NOT APP CODENếu code trong app: mỗi service phải implement retry, timeout, circuit breaker riêng → duplicate code, inconsistent behavior, khó update policy. Với Istio: thay đổi VirtualService YAML là xong — không redeploy service.
TRADE-OFFLatency overhead ~5-15ms per hop (sidecar processing). Ops complexity cao: Istio control plane, CRDs, certificate rotation. Resource overhead: mỗi pod thêm 1 Envoy container (~50MB RAM). Acceptable cho production microservices.

SM2
How do you configure traffic management in Istio — canary deploy, circuit breaker, retries?
Cấu hình traffic management trong Istio — canary, circuit breaker, retry?
›
CANARYVirtualService + DestinationRule. Ví dụ: 90% traffic → v1, 10% → v2. Tăng dần không cần redeploy. Header-based routing cho internal testing.
CIRCUIT BREAKERDestinationRule với outlierDetection: nếu service trả về 5xx liên tục → Envoy eject khỏi load balancer pool. Auto recover sau 30s. Không cần Hystrix hay Resilience4j.
RETRYVirtualService retries: max 3 attempts, timeout 2s, retry on 503/504. Idempotency quan trọng: chỉ retry GET và idempotent POST. Dùng idempotency key cho payment.
